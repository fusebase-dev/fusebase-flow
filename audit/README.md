# `audit/` — public-template audit folder

This folder is **not** for build-time phase reports. It is intentionally minimal in the public template.

## What lives here (and why)

| File | Purpose |
|---|---|
| `README.md` | This file. |
| `skill-mirror-manifest.txt` | SHA-256 manifest of skill mirrors under `.agents/skills/` and `.claude/skills/`. Tracked because `preflight.sh` and the GitHub Action use it to detect drift between canonical skills and provider mirrors. Regenerated by `bash hooks/local/mirror-skills.sh`. |
| `cli-vendor-manifest.json` | Provenance for the **vendored FuseBase CLI-owned assets** shipped in this edition (FuseBase CLI 0.25.9: 20 provider skills incl. `references/` × 2 mirrors, 2 app-agents × 2 providers, 4 `.claude/hooks/` files): per-file sha256, bundling date (`generated_at`), and `source_cli_version` (the literal `unknown` sentinel — UNVERIFIABLE_LOCALLY; freshness is advisory only). The per-file sha256 is the source `check-cli-flow-conflicts.sh` hashes against to emit the advisory `CLI_SNAPSHOT_STALE` finding. Regenerated by `bash hooks/local/stamp-cli-provenance.sh`. Distinct from `skill-mirror-manifest.txt`, which tracks only the 32 canonical **Flow** skill mirrors. |

## What does NOT live here

- **Build-time phase reports.** Phase reviews, packaging reports, and other internal build records are kept outside the public template (with the release-audit bundle that accompanies a tagged release).
- **Generated runtime reports.** Output from `install.sh`, `run-tests.sh`, and other local scripts is written to `state/audit/` (gitignored). It does not get committed.
- **CI workflow logs.** GitHub Action runs upload `state/audit/` as a workflow artifact named `fusebase-flow-audit`. Per-run logs and reports live in the artifact, not in the repository.

## Why the separation

Fusebase Flow is a GitHub template. Anything committed here is copied into every new user repository that's created from the template. Build-time audit reports and per-run test logs would clutter every consumer's tree without serving them. They belong in the release-audit bundle (tagged with the release version) and in CI workflow artifacts, not in the template itself.
