# Architect sub-agent (architect-sub-agent)

**Status:** parked
**Filed:** 2026-05-30 · **Refreshed:** 2026-06-10 (v3.16.0 baseline)
**One-liner:** Add a dedicated Architect sub-agent (alongside Product Owner + AI Developer) for deep cross-cutting investigation, so escalation tickets don't bloat the PO's context.

## Operator pain (in their words)

Today, when an investigation surface exceeds 10 files, hits a cross-cutting refactor, or surfaces a blocked-migration design, the Product Owner sub-agent **absorbs** Architect duties inline — applying `AR.1..AR.7` rules additively. This works for most escalation cases but means a single PO session carries both the day-to-day spec/decision/task drafting *and* deep investigation in the same role context. On larger refactors, a dedicated Architect identity (separate tool surface, deeper investigation toolbelt, fresh session) would let the PO stay lean while the Architect carries the heavy investigation.

## Why now

The `role-discipline` skill already has a fully-shaped `Architect (escalation)` section with don't-list + refusal phrasing — the discipline exists; only the *separate sub-agent definition* is missing. The framework line is stable (v3.16.0: 25 FR rules, 28 skills) and `workflows/architect-escalation.md` already describes the escalation procedure the agent would own.

## Architectural sketch (rough)

- New canonical at `agents/architect/AGENT.md` (alongside `agents/product-owner/`, `agents/ai-developer/`).
- Tool surface: similar to PO (no app-code edits) but with broader read/grep allowance and access to `docs/constitution.md` / `policies/protected-paths.yml` for migration-constraint cross-checking.
- Invocation: explicit operator action — *"Use the architect sub-agent. Investigate <area>."* — returns a recommendation to PO who locks decisions.
- Self-attestation: standard FLOW_RULES self-attestation naming the Architect role + the role-discipline AR section.
- PO continues to handle non-escalation tickets entirely; PO still applies AR rules inline for *minor* escalations (the dedicated Architect is for *substantial* investigation surfaces).
- Mirrored via `hooks/local/mirror-agents.sh` to `.claude/agents/architect.md` and `.codex/agents/architect.md`.
- README capability catalog: sub-agent count 2 → 3.
- Architect findings relay: formal output as `docs/tmp/handoff/<date>-<slug>-architect.md` (template `templates/architect-response.md` already exists).

## Acceptance criteria (rough)

3. AC3 — PO sub-agent's "Escalation triggers" section gains: *"For substantial investigation surfaces (≥20 files / multi-week refactor), hand off to the Architect sub-agent rather than running AR rules inline."*
4. AC4 — `workflows/architect-escalation.md` updated to describe both modes: PO-inline (small escalations) and Architect-sub-agent (substantial escalations).
5. AC5 — README capability catalog reflects 3 sub-agents.
6. AC6 — Preflight 0/0 + hook tests all-pass (22/22 at the v3.16.0 baseline); no FR-rule changes.

## Out of scope

- New FR rule — Architect already has its AR section in role-discipline; no new framework rule needed.
- Hard-fork of PO duties — PO still absorbs Architect for small escalations; the new sub-agent is opt-in for big ones.
- Auto-escalation routing — the operator decides whether to invoke Architect.

## Risks / unknowns

- **Boundary ambiguity** — "when does the PO escalate to Architect vs. handle inline" needs sharp criteria (e.g., investigation surface ≥ 20 files OR blocked migration OR cross-subsystem refactor spanning ≥ 3 modules); otherwise operators over- or under-escalate.
- **Three sub-agents may be too many** — ship opt-in; PO-inline stays the default. If operators don't use it, it's a no-op.
- **Architect handoff format** — `templates/architect-response.md` exists; confirm it fits the `docs/tmp/handoff/<date>-<slug>-architect.md` relay shape (v3.13.0 path model) before reuse.
