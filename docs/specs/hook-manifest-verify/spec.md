# Spec — hook-layer manifest verification + single-process test runner (DRAFT, v4)

Ticket: hook-manifest-verify · Lane: Full · Status: DRAFT (v4 — Codex-vetted design
+ implementation; AC3 resolved via platform-adaptive `--run-hook-tests`. Finding-closure
maps in decisions.md § Revision log)

## Problem

Full `fusebase-flow-health-check.sh` cannot reach HEALTHY on Windows/Git-Bash: the
hook-test CRITICAL runs `hooks/tests/run-tests.sh`, a fork-per-case suite (21 fixture
runs × 2+ spawns + ~20 fork-heavy bash phases ⇒ thousands of spawns). MSYS spawn cost
(~0.8–1.4 s vs ~1–3 ms POSIX) makes the suite 7–8 min with 240 s INCONCLUSIVE risk, so
Windows operators are capped at `--fast` ⇒ PARTIAL_UNVERIFIED / exit 4 forever.
Root defect class: the verdict re-derives test correctness on every consumer host,
when correctness was already proven once at ship time.

## Solution (two parts)

**Part 1 — prove at ship time, verify integrity at install time.**
1. CI (ubuntu) runs the FULL hook-test suite on every push/PR to main
   (`fusebase-flow-verify`, now also callable via `workflow_call`). Publication is
   GATED in-repo (v3/B4): a `v*` tag push triggers
   `.github/workflows/fusebase-flow-release.yml`, whose `publish` job — the only
   thing that creates the GitHub Release — declares `needs: verify`, where `verify`
   runs the same full suite + runner parity + manifest freshness on the tagged sha.
   A red suite ⇒ `publish` never runs ⇒ no Release exists for that tag (D10).
   Repo-admin backstops (a `v*` tag ruleset + `main` branch protection requiring
   the `verify` check) are documented WITH apply commands in PUBLISHING.md (D10.4)
   — they close the raw-tag-ref and manual-bypass surfaces that files cannot.
2. Every release carries a committed content-hash manifest of the Flow-owned hook
   layer: `audit/hook-layer-manifest.json`, generated by
   `hooks/local/stamp-hook-manifest.sh` (pipeline-regenerated, never hand-edited;
   CI fails if it is stale). Stamping is **byte-stable**: the manifest contains no
   timestamps (`generated_at` deliberately absent — D1), so
   `stamp && git diff --exit-code` is deterministic forever, not only on the commit
   date.
3. The health check's hook-test CRITICAL becomes **manifest verification**: one python
   pass hashing ~70 files (seconds, OS-independent). Match ⇒ ✓ + full HEALTHY reachable
   on all OSes. Mismatch ⇒ exact drifted files reported (also catches local tampering,
   which re-running tests does NOT detect — tampered code can pass tests). The
   verifier also flags injected extras: unexpected `hooks/handlers/*.py` /
   `hooks/shared/*.py` files AND any `sitecustomize.py`/`usercustomize.py` at any
   depth under `hooks/` (mirrors policies/protected-paths.yml:93–100; D3).
4. Local test execution stays available as `--run-hook-tests` — an optional deep
   diagnostic that runs the FULL `run-tests.sh` suite (never required for the
   verdict).

**Part 2 — make the local suite MSYS-viable WITHOUT losing coverage.**
- The default `bash hooks/tests/run-tests.sh` remains the FULL unscoped local gate:
  every phase that runs today keeps running (D7). Exactly two composition changes:
  1. The fixture fork-loop is REPLACED by a single-process runner
     (`hooks/tests/run_hook_tests.py` imports each handler and drives `main()`
     in-process; handlers/shared unmodified — FR-07). Same 21 fixtures, same
     assertions, same PASS/FAIL output shapes. A `--compare-subprocess` mode proves
     in-process ≡ subprocess for **(exit_code, decision, rule_id)** on all fixtures.
  2. Two phases are ADDED: a single-pass git-wrapper smoke (`git-smoke`, D9) and a
     stamp/verify self-test (`hook-manifest`, T5).
- There are NO tiers (`--full`/`--core` rejected — D7). Fast iteration = the existing
  explicit `FF_ONLY` scoping, which is fail-closed and clearly labeled
  not-a-full-gate by construction.
- The MSYS wall-time budget (< 120 s for the full suite) is met empirically via the
  D14 measure-then-optimize protocol: per-phase timing instrumentation, measurement
  on a real Win11/Git-Bash box, and named optimization levers for the known-heavy
  bash phases (cli-flow-recovery, cli-0259, secret-scan-staged, bootstrap-exception,
  ws5-upgrade, sync-allowlist, bootstrap-baseline-hop).

| `hooks/local/lib/*` (files: `*.sh` + the manifest lib `*.py`) | — |

(consumer-merged project rows / `*.local.yml` layering ⇒ false drift),
`hooks/local/fusebase-flow-overlays/**` (recovery templates, separate ownership
mechanism), `hooks/requirements.txt`, the manifest file itself (no fixed point —
protected by self-hash + commit + CI freshness).

Byte-determinism prerequisite: `.gitattributes` must pin `*.jsonl text eol=lf`
(currently unpinned ⇒ `core.autocrlf=true` Windows checkouts would CRLF-rewrite the
fixture transcripts ⇒ guaranteed false drift). All other covered files are already
pinned (`*.py/*.sh/*.json` + explicit `hooks/git/pre-commit|commit-msg`).

## Trust model (document of record)

- **Manifest** attests: *this install's Flow hook layer byte-matches release X.*
- **CI** attests: *release X passes its full hook-test suite on Linux* — the
  verify job runs on main pushes and PRs, and the tag-triggered release workflow
  re-runs it on the tagged sha with the Release-creating `publish` job gated
  behind it (`needs: verify`): a red suite means the GitHub Release is never
  published (D10, AC4). The raw tag ref and a manual-release bypass are closed by
  documented repo-admin settings (tag ruleset + branch protection, PUBLISHING.md),
  not by files — boundary stated honestly in D10.4.
- Consumer verification is a **membership proof** (hash compare), not re-derivation of
  correctness. CI ordering guarantees the committed manifest is only ever current for
  a tree whose full suite is green (tests step precedes the manifest-freshness step;
  any red step fails the workflow).
- **Injection surface:** beyond hash-matching listed files, verify flags UNEXPECTED
  extras on the import-adjacent surface (`hooks/handlers/*.py`, `hooks/shared/*.py`)
  and any python startup file (`sitecustomize.py` / `usercustomize.py`) at ANY depth
  under `hooks/` — the same names policies/protected-paths.yml:93–100 protects
  (defense-in-depth alongside the `python3 -S` runtime close).
- **Self-hash** (`manifest_self_sha256` over the canonical payload) detects
  corruption and hand-edits of the manifest. It is NOT a cryptographic defense
  against a determined attacker (who can recompute it); that backstop is the
  committed manifest + git history + the CI freshness gate. A local attacker who
  tampers the verifier/engine itself is out of threat model — consistent with the
  existing `ffhc_select_pass_line` threat-model note (harness control ⇒ repo already
  compromised).

## Acceptance criteria

| # | Criterion |
|---|---|
| AC1 | Full `fusebase-flow-health-check.sh` (no `--fast`) completes < 60 s and returns HEALTHY / exit 0 on stock Win11 + Git-Bash when the tree matches the manifest. |
| AC2 | Tamper one byte in a covered `hooks/shared/*.py` (uncommitted, reverted after) ⇒ health check exits 1 / FLOW_LAYER_DRIFT naming THAT file. Corrupt the manifest self-hash ⇒ exit 2 / BROKEN. Manifest absent ⇒ standalone verifier exit 4 and health check exit 4 / PARTIAL_UNVERIFIED (never a crash) — matches the check-cli-flow-conflicts absent-provenance precedent. (Verifier exit 3 is reserved-unused; the engine's public exit 3 = EXCEPTION_IN_EFFECT is untouched.) |
| AC3 | `--run-hook-tests` is PLATFORM-ADAPTIVE: on Linux/POSIX it runs the FULL `run-tests.sh` suite (the same suite as before); on MSYS/Git-Bash it runs the FAST single-process path (the fixtures runner + git-smoke + hook-manifest phases), completing HEALTHY (rc 0) in < 120 s wall — measured 52 s on a real Win11/Git-Bash box. The FULL suite on MSYS stays reachable via `--run-hook-tests-full` / `FFHC_RUN_HOOK_TESTS_FULL=1`. The DEFAULT `run-tests.sh` keeps full coverage on all platforms; CI (Linux) is the full-suite authority. The base health verdict is never gated on the optional deep run. (v4: supersedes the v3 "full suite on all platforms" wording — D14.6 measured the full MSYS suite at ~950 s, dominated by non-fork-storm phases outside this ticket's levers; D14.8 re-locked the platform-adaptive resolution.) |
| AC4 | Release publication is GATED in-repo: `.github/workflows/fusebase-flow-release.yml` triggers on `v*` tag push; its `publish` job — the ONLY `gh release create` under `.github/` (the manual step is removed from PUBLISHING.md and forbidden) — declares `needs: verify`, where `verify` calls the full verify workflow (full tests → runner parity → manifest freshness, in that order) via `workflow_call`. A red hook test ⇒ verify fails ⇒ publish is skipped ⇒ no GitHub Release exists for that tag; a stale committed manifest fails the same way. PUBLISHING.md § Release prerequisites documents the repo-admin backstops (`v*` tag ruleset + `main` branch protection requiring the `verify` check, each with its `gh api` apply command) and states honestly which surfaces only they can close (raw tag ref; manual bypass). |
| AC5 | `bash hooks/local/upgrade.sh` on an existing consumer picks up the manifest + stamp/verify scripts + updated health check + runner with no manual steps. |
| AC6 | Back-compat: `--fast` unchanged (PARTIAL, exit 4, never 0); verdict names and engine exit codes 0/1/2/3/4 unchanged; strict `[run-tests] N/N PASS` + `exit $fail` contract unchanged; the default `run-tests.sh` phase set is a strict SUPERSET of today's (nothing moves behind a flag); Linux/macOS no regression. |
| AC7 | Parity: `run_hook_tests.py --compare-subprocess` shows identical **(exit_code, decision, rule_id)** per fixture, 21/21, on MSYS and in CI. |

## Out of scope

- Extending the manifest to `policies/`, overlays, adapters, or provider mirrors.
- Cryptographic signing of the manifest.
- BEHAVIOR changes to the retained bash scenario tests — D14 permits
  behavior-preserving performance consolidation only (same scenario names + same
  assertions).
- `post-fusebase-update.sh` changes (upgrade.sh is the propagation path, AC5).
- Version bump / release execution (Deploy phase, operator-owned). Repo-settings
  changes (the `v*` tag ruleset, branch protection) are operator actions documented
  in PUBLISHING.md, not performed by the implementer.
