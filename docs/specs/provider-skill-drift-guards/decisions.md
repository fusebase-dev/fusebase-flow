## B1. Vendoring vs reference-over-copy (Issue [4])

**Recommendation:** Keep vendoring the CLI provider assets; close the drift risk with provenance + freshness + drift-aware checks rather than de-vendoring.

**Reasoning:** Reference-over-copy is the cleaner root-cause fix, but the "Fusebase CLI edition" value proposition is precisely that it ships the CLI domain assets so the template works offline / before `fusebase update` runs (README.md "Edition scope"; `docs/fusebase-cli-edition.md`). De-vendoring would change install UX and remove that guarantee. The verified baseline (canonical-only mirror, `flow_write_mode:"never"`, recovery exclusion — all CONFIRMED holding) already prevents Flow from *writing* CLI assets; the residual risk is *staleness visibility*, which provenance + advisory freshness addresses without abandoning the model.

**Alternatives considered:**

- **Reference-over-copy (de-vendor):** rejected — breaks offline/template UX; larger blast radius; contradicts edition intent.
- **Minimal/correctness-only (accept 1–3 as by-design):** rejected — leaves the present-but-stale blind spot and the install-overwrite vector open.

**Lock status:** LOCKED

## B2. Provenance mechanism

**Recommendation:** Add a separate read-only manifest `audit/cli-vendor-manifest.json` (source-CLI version | `unknown`, bundling date, per-file sha256) covering all vendored CLI-owned assets, regenerated by `hooks/local/stamp-cli-provenance.sh`. Do NOT fold CLI assets into `audit/skill-mirror-manifest.txt` (that file is regenerated from canonical Flow skills by `mirror-skills.sh` and must keep tracking only the 14 Flow skills). Do NOT stamp 19+ skill frontmatters individually.

**Reasoning:** A single manifest is the least-invasive provenance surface, leaves the canonical-only mirror manifest untouched (no regression of the verified baseline), and gives the checker (B3) and CI a hash source. Covers the critic's A1 (agents) and A2 (hooks) gaps, not just skills.

**Alternatives considered:**

- **Per-file frontmatter stamps:** rejected — touches 19+ files, risks merge churn on every `fusebase update`, and the SKILL.md entrypoints would still need a parallel manifest for `references/`.
- **Extend `skill-mirror-manifest.txt`:** rejected — `mirror-skills.sh` rewrites it from canonical Flow skills; CLI entries would be erased on next mirror.

**Lock status:** LOCKED

## B3. Drift detection in conflict reporter

**Recommendation:** Extend `check-cli-flow-conflicts.sh` to (a) hash each present CLI asset against `cli-vendor-manifest.json` and emit advisory `CLI_SNAPSHOT_STALE` (info, non-failing) when changed; (b) scan CLI-owned skill files for `CUSTOM:SKILL` blocks and report them "at-risk on next refresh". Keep `MISSING → CLI_LAYER_DRIFT` as-is. Restoration stays CLI-owned (advisory points to `fusebase update`).

**Reasoning:** Closes Issue [2] (existence-only) and Issue [3] (CUSTOM blocks) without changing the CLI-first/Flow-second posture. Advisory-not-failing keeps verdict semantics stable and avoids false BROKEN on legitimate CLI updates.

**Alternatives considered:**

- **Rename to "ownership reporter", drift out of scope:** rejected — the tool is named a *conflict* reporter; users reasonably expect currency, not just presence.
- **Make stale a hard failure:** rejected — a newer CLI copy is expected and correct; failing on it inverts the model.

**Lock status:** LOCKED

## B4. CLI app-agent ownership

**Recommendation:** Replace the `.claude/agents/app-*.md` and `.codex/agents/app-*.md` wildcard entries in `agent-surface-ownership.json` with explicit `known_names: ["app-architect","app-create-checker"]`, mirroring the skill pattern; update the checker's agent branch to iterate `known_names` instead of `glob("app-*.md")`.

**Reasoning:** Confirmed Issue [6]/A5: the wildcard is consumed as a live glob (`check-cli-flow-conflicts.sh:259-267`) and would silently misattribute a future Flow agent named `app-*` as cli-owned/`never`. Explicit names match the already-correct skill approach and remove the asymmetry.

**Alternatives considered:**

- **Keep glob, add a Flow-name exclusion list:** rejected — two lists to maintain; explicit allow-list is simpler and consistent with skills.

**Lock status:** LOCKED

## B5. Stop-hook consolidation

**Recommendation:** Standardize the example Stop hooks on the node implementations (`run-typecheck-apps.js`, which carries the CVE-2024-27980 `shell:win32` patch, plus the node lint path in `quality-check-apps.js`). Update `.claude/settings.json.example` to wire node hooks; unwire and deprecate the jq/bash duplicates (`run-typecheck-on-stop.sh`, `run-lint-on-stop.sh`). AI Developer to confirm `quality-check-apps.js` coverage before deleting vs deprecating.

**Reasoning:** Confirmed Issues [7]/[8]: the wired jq/bash hooks fail out-of-the-box on Windows (no jq/bash) while the Windows-safe node hook is idle — a self-contradiction. Node hooks are cross-platform and already patched.

**Alternatives considered:**

- **Add jq/bash to prerequisites:** rejected — pushes a Windows-hostile dependency; the node path already exists.
- **Delete the .sh immediately:** allowed only if `quality-check-apps.js` fully covers lint+typecheck; otherwise deprecate-with-header for one release.

**Lock status:** LOCKED

## B6. Install-overwrite hazard

**Recommendation:** Change the documented install copy steps so CLI-owned paths copy **only if absent** (or are explicitly excluded from the recursive copy / `-Force`), while Flow-owned paths copy normally. Document the two-writer (`fusebase update` vs Flow snapshot) hazard, including `CUSTOM:SKILL` blocks, in `docs/fusebase-cli-edition.md`.

**Reasoning:** Critic A3 (HIGH): `docs/install-existing-project.md:80/101` does `cp -R .../.claude .` / PowerShell `-Force`, wholesale-overwriting live CLI skills+hooks+agents with the frozen snapshot — directly violating the repo's own "Flow must never restore CLI assets from its bundled copy" rule. This is the strongest concrete data-loss vector and is currently guarded only by prose.

**Alternatives considered:**

- **Prose-warning only (status quo):** rejected — the copy block itself has no guard; `-Force` overwrites unconditionally.

**Lock status:** LOCKED
