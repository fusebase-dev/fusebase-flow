## Problem

Implementation sessions consume tokens without considering efficiency — the uncovered leak classes after FR-21/22/23/25 + v3.19.1: read-side waste (re-reading unchanged files, whole-file reads for one fact, reading generated/vendored files, re-deriving known IDs), retry storms (same failing approach re-attempted instead of diagnosed), whole-file regeneration for small edits, and — root cause — **no measurement**: nobody can see where a session's tokens went. Constraint (operator): must not damage product quality, development depth, or LLM intelligence.
