# Fusebase Flow v3.2.0 — provider-skill drift guards

**Release date:** 2026-05-29
**Previous version:** 3.1

## Summary

The Fusebase CLI edition vendors a second copy of FuseBase CLI-owned assets (19 provider skills plus their `references/`, 2 CLI app-agents, 4 Claude Code quality hooks) inside the Flow template so the edition works offline / before `fusebase update` runs. Those vendored copies are written by **two independent tools** — `fusebase update` (the live CLI bundle) and the frozen Flow snapshot — with no provenance, no freshness signal, and no content-drift detection.

The recovery model already prevented Flow from *writing* CLI assets (mirror scope is canonical-only; the 19 CLI skills are `flow_write_mode:"never"`; `post-fusebase-update.sh` excludes CLI paths — all verified holding). v3.2.0 closes the residual gaps: no provenance/version stamp, an existence-only conflict reporter, an undetected two-writer overwrite hazard on install, a fragile `app-*.md` agent wildcard, and Windows-fragile jq/bash Stop hooks wired while the Windows-hardened node hook sat idle.

De-vendoring (reference-over-copy) was considered and **rejected** (decision B1): it would break the offline/template UX that is the edition's value proposition. Vendoring stays, gated behind provenance + freshness + drift-aware checks.

## Added

### Provenance manifest + generator (B2)

- New `hooks/local/stamp-cli-provenance.sh` enumerates every vendored CLI-owned asset and writes `audit/cli-vendor-manifest.json`:
  - `{ schema_version, generated_at, source_cli_version, asset_count, assets: [{ path, sha256 }] }`
  - skill/agent name lists are driven from `agent-surface-ownership.json` `known_names` (no second hand-maintained list).
  - `source_cli_version` is the literal `"unknown"` sentinel — the bundling tool cannot know which live CLI bundle a copy came from (UNVERIFIABLE_LOCALLY). Freshness is advisory only.
  - read-only / idempotent; the gitignored local-only `repo-polish` skill is excluded.
- The manifest is a **committed** document of record (like `audit/skill-mirror-manifest.txt`), not gitignored.
- `preflight.sh` gains an advisory (non-failing) check that the manifest exists and parses.

### Drift-aware conflict reporter + CUSTOM scan (B3)

`hooks/local/check-cli-flow-conflicts.sh` now:

- hashes each **present** CLI-owned asset against the provenance manifest and emits an advisory `CLI_SNAPSHOT_STALE` finding when the sha256 differs (a newer or locally-modified copy);
- scans CLI-owned skills for `<!-- CUSTOM:SKILL:BEGIN -->…END` blocks and reports any as `CLI_CUSTOM_AT_RISK` ("at-risk on next refresh").

Both signals are **informational only**: they are excluded from `cli_drift`, so they never flip the verdict away from `HEALTHY` or change the exit code. `MISSING → CLI_LAYER_DRIFT` (exit 1) semantics are unchanged. An absent/invalid provenance manifest simply skips the advisory; it never fails. Restoration of CLI content always stays with the CLI.

## Out of scope (deliberately)

- **De-vendoring / reference-over-copy** — rejected (B1); preserves offline/template UX.
- **Restoring CLI-owned content from the Flow snapshot** — remains CLI-owned; Flow diagnoses only.
- **Comparing the bundled snapshot against the *live* CLI bundle** — UNVERIFIABLE_LOCALLY; `source_cli_version` is the `unknown` sentinel and freshness is advisory only.

## Baseline protections (re-verified non-regressed)

- `mirror-skills.sh` iterates canonical `skills/` only (14 Flow skills); CLI skills are never written by it.
- All 19 CLI provider skills keep `flow_write_mode:"never"`.
- `post-fusebase-update.sh` restore set still excludes `.claude/hooks/**`, CLI provider skills, MCP / `fusebase.json` / `skills-lock.json`, and active `.codex/config.toml`.
- `audit/skill-mirror-manifest.txt` still 28 lines (14 Flow skills × 2 mirrors); CLI skills excluded.

## Compatibility

Existing projects adopt v3.2.0 by pulling the updated framework files and regenerating mirrors + provenance:

```bash
bash hooks/local/mirror-skills.sh
bash hooks/local/mirror-agents.sh
bash hooks/local/stamp-cli-provenance.sh
bash hooks/local/preflight.sh
bash hooks/tests/run-tests.sh
bash hooks/tests/test-cli-flow-recovery.sh
bash hooks/local/check-cli-flow-conflicts.sh
```

No per-ticket artifact migration is required. The provider surface still contains 33 skills total (14 Flow mirrors + 19 CLI provider skills); the Flow mirror manifest tracks only the 14 canonical Flow skills, and the new `cli-vendor-manifest.json` tracks the vendored CLI assets.
