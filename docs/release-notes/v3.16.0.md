# Fusebase Flow v3.16.0 — FR-25 module-size ratchet

**Release date:** 2026-06-10
**Previous version:** 3.15.0

## Summary

Closes the framework's **structural blind spot**: nothing in the lifecycle ever looked at file size. A consumer audit (paperclip+hermes-v1) found source files of **19,026 / 14,202 / 10,434 / 5,363 lines** accreted under full Flow discipline — every individual diff passed review; the monolith is the integral of N reasonable diffs. Root causes are framework-level: tasks say WHAT but never WHERE; every gate is behavioral; one-task-one-commit + FR-21 make mid-task extraction look like scope creep.

Source is AI-read (FR-22/FR-24 audience principle): a 19k-line file can't be loaded in one pass, so every future session pays degraded slice-reads on the hottest files.

## The key asymmetry

Unlike FR-22 (comments) and FR-23 (doc budget) — semantic, deliberately gate-less — **line count is objective**. FR-25 is the first write-time rule that ships a **deterministic gate**, not just steering prose. Split *quality* (responsibility seam vs mechanical `utils2.ts`) stays semantic → review-time only.

## Adoption (downstream)

1. Upgrade normally (`hooks/local/upgrade.sh`). The gate runs **warn-only** out of the box.
2. Add justified monolith classes (upstream mirrors, data-as-code catalogs) to `exempt_globs` — or to `policies/module-size.local.yml`.
3. Activate: `bash hooks/local/check-module-size.sh --write-baseline` → commit `policies/module-size-baseline.txt`. Existing monoliths freeze at current size; new growth blocks.

## Not done (deliberately)

- No forced refactor of existing monoliths (decompose via explicit tickets — ratchet stops the bleeding).
- No regex gate for split quality (semantic — `code-review` owns it).
- No `*.md` gating (FR-23 owns docs). No template-shipped baseline (would block legacy-repo installs on first touch).
- `docs/rail-mapping.md` rows still end at FR-19 (pre-existing documented gap, operator-tracked) — not extended here.
