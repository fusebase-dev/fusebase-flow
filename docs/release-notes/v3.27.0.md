# FuseBase Flow v3.27.0 — FR-22 delivery guarantee + PO verifiable boot

**Release date:** 2026-06-17
**Previous version:** 3.26.0
**Type:** MINOR (feature — additive)

## Summary

Two tickets in the **FR-22 family** — the lesson that *"mandatory" without a delivery guarantee and artifact-level verification is just "optional with extra words."* Both are **warn-only** (no deny gate loosened) and **FR-07-clean** (no FR rule rows, no deploy-policy rule semantics, no `ratchet-governance.yml`). Both were independently **Codex-SHIP'd** (design RESCOPE folded + implementation review). No new FR rule and no new skill — the FR range (FR-01..FR-26) and skill count (31) are unchanged.

1. **FR-22 delivery guarantee** (consumer report: "mandatory in prose, undelivered in practice"). The comment-policy is now **present-by-construction** in code-writing handoffs, the write-time digest states the rule **does not auto-propagate to sub-agents**, and a warn-only `comment_policy_review_applied` signal ends the loop's comment-blindness. Every piece is **artifact-level** (process), never gating on comment **content** — FR-22's no-content-gate stays correct and intact.
2. **PO verifiable boot**. `/product-owner` now opens with an operating-rules checklist + a **machine-detectable `PO-ACTIVATED` marker** (delivered by construction via the command **and** the canonical agent), with a **warn-only Stop-hook verification** riding the already-wired `UserPromptSubmit` / `Stop` events — the firing point that the deferred FR-22 hook F lacked.

## What ships

### FR-22 delivery guarantee (Phase 1 — A–E)

| Piece | What it does |
|---|---|
| **Present-by-construction handoff block** (`templates/handoff-implement.md`) | The FR-22 Delegation push-block text is now a literal, non-optional section in every authored implement handoff — not a "remember to inline" instruction that a long autonomous run forgets. |
| **Write-time digest carries the full rule + non-propagation note** (`flow-skills/role-discipline/SKILL.md`) | The digest now states the full two-kinds FR-22 rule **and** that it does **not** auto-propagate to sub-agents — when delegating code-writing, inline the comment-policy Delegation push block. |
| **`comment_policy_review_applied` warn-signal** (`policies/required-artifacts.yml` + `hooks/handlers/stop.py`) | A transcript marker recording that the comment review **ran** (`comment-policy review: applied (FR-22)` or `… N/A (FR-22; no code diff)`); absence → **warn**, never block. Records that the review ran, never what comments say (FR-22-safe). |
| **Artifact-vs-content distinction doc** (`flow-skills/comment-policy/SKILL.md` + `docs/comment-policy.md`) | States explicitly: gating on comment **content** is forbidden, but artifact-level checks (handoff-contains-block, review-ran-signal) are **encouraged** — so maintainers stop conflating the two and building nothing. |
| **Carve-out clarity** (`policies/comment-policy.yml`) | Opt-in posture preserved (local-override-replaces-default, like `protected-paths.yml`); clearer "derive at Specify via `references/audit-prompt.md`" trigger + a tighter commented starter set. No gate. |

**Deferred:** `fr22-predelegation-hook` (the pre-delegation PreToolUse check) — design review found it **inert under the shipped matchers** (they match Bash/Edit/Write only, not the Task/Agent delegation tool), so as written it would recreate the mandatory-but-undelivered failure it fixes. Its own ticket: needs host-matcher coverage, explicit delegation markers, and warn-only telemetry first.

### PO verifiable boot

| Piece | What it does |
|---|---|
| **Command boot block + ASCII marker** (`.claude/commands/product-owner.md` + overlay source) | `/product-owner` renders a 6–8 line operating-rules checklist the PO echoes as its first reply, ending in the ASCII `[[ PO-ACTIVATED \| FuseBase Flow <VERSION> \| … ]]` marker. Pointers to FLOW_RULES, never a re-paste. |
| **Same boot in the canonical agent** (`agents/product-owner/AGENT.md` → mirror) | Carries the activation block + marker so the PO boots-by-construction whether reached via the slash command or spawned via the Agent tool / description match. A drift-guard test asserts the command and agent boot blocks match. |
| **UserPromptSubmit detection** (`hooks/handlers/user_prompt_submit.py`) | Detects a `/product-owner` invocation → one-line non-blocking reminder + a supplemental `po_activation_requested` audit event. Warn-class, never blocks. |
| **Dedicated PO-activation Stop check** (`hooks/handlers/stop.py`) | A **separate** path outside `CLAIM_PATTERNS` (so it fires on a PO first reply, which has no done/deploy claim): PO activation present but marker absent → **warn + Stop ALLOWS**. Literal transcript scan only; never inspects PO reasoning; existing deny gates untouched. |

## Why the PO boot is enforceable where FR-22 hook F was not

FR-22's pre-delegation hook needed a **Task-tool** PreToolUse matcher the shipped `settings.json.example` does not include → inert. The PO boot rides on **`UserPromptSubmit`** (detect `/product-owner`) + **`Stop`** (detect the marker) — both are **general events already wired** in the example settings. No new matcher coverage required.
