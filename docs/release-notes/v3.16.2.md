# Fusebase Flow v3.16.2 — FR-25 hardening (gate live by default)

**Release date:** 2026-06-10
**Previous version:** 3.16.1

## Summary

Closes the delivery gaps a post-ship stress test found in FR-25. Evidence first: an empirical probe of the motivating consumer repo showed its monoliths **still growing days after the audit** (workflow.ts 14,202→15,616; helpers.ts 10,434→10,840; 45 of 399 source files over ceiling; 58% of all source lines in over-ceiling files) — steering alone does not stop growth. An independent devil's-advocate review of the v3.16.0 design concluded "right call, wrong delivery posture": the ratchet core held, but the deterministic layer shipped dormant. No rule text changes.

## The decisive fix

GitHub-template distribution is **copy-once** — defaults are ~permanent per instantiation, and the median consumer is greenfield. v3.16.0 shipped no baseline (optimizing for retrofit safety), which made the gate silent exactly where prevention is cheapest. v3.16.2 ships the template's own committed baseline (1 dogfood row), so **the gate blocks from commit #1 on greenfield repos**; retrofit installs re-key once (`--write-baseline`, now a documented install step, also printed by every block message). This supersedes the v3.16.0 M4 decision with the install-path nuance both cases need.

## Known accepted (from the stress test, deliberately not done)

- No per-file growth slack for in-place edits to frozen monoliths — the targeted re-key is the operator-mediated remedy; slack would soften the ratchet for everyone to ease one case.
- No automatic baseline carry across renames (radar item; `--write-baseline <new-path>` is the one-command remedy).
- No directory-sprawl / total-retrieval-cost instrument — different metric, future code-map ticket (ROADMAP radar).

## Verified

- test-module-size.sh 8/8; run-tests **24/24**; preflight 0/0
- `check-module-size.sh --all` green against the shipped baseline
- S7 live proof: local `enforcement: warn` present → violation still exits 1
