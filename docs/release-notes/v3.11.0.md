# Fusebase Flow v3.11.0 — FR-22 write-time delivery

**Release date:** 2026-06-06
**Previous version:** 3.10.0

## Summary

Closes the **delivery gap** in FR-22 (the code-comment policy shipped in v3.10.0). FR-22 shipped as a correct *rule* but had no **write-time carrier** — its body never reached a code-*writing* agent's context at the moment comments are written. A v3.10.0 consumer (`WorkHub Managed`) proved the gap in production: an AI Developer sub-agent wrote default "explain-richly" JSDoc-heavy comments — the exact density-ratchet FR-22 was authored to break — because the breaker was never loaded. This release ships the carrier skill, fixes the false "already loaded" claim that suppressed the workaround, makes the audit prompt reachable in consumers, and adds a **sub-agent push** path after a behavioral probe (V7) showed pull (auto-load) does not reach a delegated sub-agent. FR-22's semantics are unchanged — tripwire/pointer/carve-out definitions stay verbatim.

## Why a new skill (not just the rule)

Three root causes, each delivery (not semantics):

1. **No write-time carrier.** Every other behavioral FR has a loaded skill (FR-08/09 → `communication`, FR-16–20 → `role-discipline`, FR-21 → `lightweight-lane`). FR-22's only homes were `FLOW_RULES.md` (existence-checked at bootstrap, never injected) and `code-review` (review-time, on-demand). A code-writing agent never had the rule in context while writing comments.
2. **A false "already loaded" claim** at `role-discipline/SKILL.md:50` asserted the rules are injected at session bootstrap. Untrue — the hook existence-checks, it does not inject — and an agent that trusts the claim never opens the file, so the claim actively suppressed the only workaround.
3. **The audit prompt was unreachable in consumers.** `upgrade.sh` deliberately does not copy framework `docs/` into consumers, yet FR-22 + `comment-policy.yml` pointed at `docs/comment-policy.md` to derive carve-outs — a dead path downstream.

A new **description-matched** skill (not a 3rd always-loaded mandatory skill) reaches the code-writing role and its sub-agents at zero context cost to Product Owner / Architect / Deploy sessions that never write code — honoring FR-22's own token-economy ethos.

## Behavioral proof (V7 → V8, the gold standard)

- **V7 (pull) — NEGATIVE.** A fresh, unprimed `general-purpose` sub-agent given a throwaway code task wrote ~49 comment lines (~90% removable per FR-22) — default JSDoc-heavy, indistinguishable from un-Flowed output, even with CLAUDE.md context present. The `comment-policy` skill did not auto-load for the delegated sub-agent. This drove the W4 decision.
- **V8 (push) — A/B PASS.** The same task with the Delegation push block **inlined** into the sub-agent's prompt produced tripwire+pointer-lean comments. The visible A/B delta proves push delivers where pull did not.

## Explicitly NOT done

- **No regex/lint/gate comment-matcher.** Distinguishing a tripwire from a restate-WHAT comment is semantic, not pattern-matchable; a gate would train agents to write worse comments to satisfy it. Enforcement stays write-time (the carrier skill + the rule) + review-time (`code-review`).
- **FR-22 semantics untouched.** Only delivery (the carrier, the pointers, the push) changed; tripwire/pointer/carve-out definitions are verbatim from v3.10.0.
- **Not retroactive.** Existing over-commented files in any consumer are cleaned only via an explicit Lightweight pass (comments strip from build output, so cleanups need no deploy).
- **`docs/comment-policy.md` kept** as the framework-dev rationale home (audience: the maintainer); not deleted.

## Adoption (per project)

1. Re-mirror or `fusebase update` to pick up the new `comment-policy` skill (additive — older mirrors are unaffected until re-mirror).
2. Run the audit prompt in `flow-skills/comment-policy/references/audit-prompt.md` against your repo.
3. Set the returned carve-out globs in `policies/comment-policy.yml: trust_critical_globs`.

Rollback: `git revert <SHA>` (one commit).
