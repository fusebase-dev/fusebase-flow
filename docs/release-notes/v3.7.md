# Fusebase Flow v3.7.0 ‚Äî Lightweight Lane (FR-21)

**Release date:** 2026-06-01
**Previous version:** 3.6.0

## Summary

Fusebase Flow applied the **same full lifecycle to every change**, regardless of size or risk. Production feedback measured a one-line, reversible edit running the full lane (spec ‚Üí decisions ‚Üí tasks ‚Üí gate ‚Üí two-agent build-then-deploy split + DP.1 approval artifact + DP.6 magic phrase) at **~10‚Äì16 min wall-clock ‚Äî ~98% process/build/verify/approval, ~2% the actual change**. Beyond the wasted time it had two second-order costs: **approval fatigue** (a magic-phrase production approval on a trivial reversible change dilutes the approvals that matter) and **added risk** (the build-then-deploy split caused a redundant rebuild ‚Äî more surface for mistakes than the change carried).

v3.7.0 introduces a **change-size tier** so ceremony scales with risk. Every ticket is classified **Full** or **Lightweight** at Specify. The safety floor is identical in both lanes; only the planning/traceability overhead changes. No skills removed (23 ‚Üí 24 canonical).

### Eligibility gate ‚Äî Lightweight iff ALL hold
1. Small implementation, single coherent concern (no large build, no real architectural decision ‚Äî a handful of files is fine).
2. Reversible (`git revert`/restore; no schema/data migration; no hard-to-remove dependency).
3. Clear, mechanically-verifiable acceptance.
4. No new security surface.
5. No cross-cutting / public-contract change needing a decision.
6. Root cause already understood (not a Phase-1 diagnostic).

**Any fail / large feature / genuine doubt ‚Üí Full lane.** Fail-safe: when unsure, escalate up.

## Added

### FR-21 ‚Äî ceremony proportional to change size (new always-on rule)

The durable anchor (same class as FR-17 forward-momentum, FR-20 zoom-out): scale ceremony to risk, in every session including hooks-off and non-Claude agents. Names the retained safety floor and the mid-flight promotion rule. Self-attestation range is now `FR-01..FR-21`.

### `lightweight-lane` skill (24th canonical)

Single source of truth for the eligibility gate, the change-note, the one build‚Üíverify‚Üídeploy pass, and mid-flight promotion. Referenced (DRY) by `requirements-specification` (lane classification at Specify), `validation-and-qa` (LL live-proof mode), `release-deploy-reporting` (LL deploy mode), `role-discipline` (PO.16 / IM.18 / DP.12), and both agents.

### `templates/change-note.md`

The entire planning + handoff + deploy artifact for an LL change: problem ¬∑ the change ¬∑ how it's verified ¬∑ rollback ¬∑ tier ¬∑ SHA. Inline in the commit body or `docs/changes/<date>-<slug>.md`.

## What the Lightweight Lane keeps vs drops

| | Full lane | Lightweight lane |
|---|---|---|
| Planning | spec + decisions + tasks + verification-gate | one change-note |
| Handoffs | implement + deploy | one (or inline) |
| Agent passes | build-agent ‚Üí separate deploy-agent | one pass: build ‚Üí verify ‚Üí deploy (no redundant rebuild) |
| Deploy approval | DP.1 JSON artifact + DP.6 magic phrase | one explicit plain operator go-ahead ("ship it") |
| Verification | full gate report | live-proof kept + 3-question test on the AC, 1‚Äì3 lines |
| Traceability | full counter + index + backlog | T/SHA + one ledger line |

**Kept in BOTH lanes (the actual risk controls):** live proof ¬∑ explicit operator deploy go-ahead (never auto-deploy) ¬∑ FR-07 protected-path check ¬∑ documented rollback ¬∑ one commit + SHA + lint/typecheck.

## role-discipline

- **PO.16** ‚Äî classify tier at Specify; don't full-lane a trivial change or lightweight-lane a risky one; promote mid-flight; never drop the floor.
- **IM.18** ‚Äî on an LL ticket, one build‚Üíverify‚Üídeploy pass; keep the floor; STOP and promote if it grows.
- **DP.12** ‚Äî for LL, a plain go-ahead replaces DP.1 + DP.6 (DP.1/DP.6 scoped to Full lane); never auto-deploy.

## Tier-aware hook layer (opt-in ‚Äî off by default since v3.6.0)

- `policies/approval-policy.yml`: new `lightweight_deploy` operation ‚Äî a **one-command stamp** (`approve-local.sh lightweight_deploy <slug> 'ship it'`) of the operator's plain go-ahead. Friction removed (no magic phrase, no hand-authored JSON), human-in-the-loop + tier telemetry kept.
- `policies/required-artifacts.yml`: `before_deploy_command` accepts `production_deploy` **OR** `lightweight_deploy` (deploy handoff waived for LL); `before_deploy_complete_claim` waives the Full-lane-only signals (probes table, post-deploy docs commit, smoke) for LL while keeping the safety-floor signals (deploy hash + rollback).
- `hooks/handlers/stop.py`: tier-aware via a lightweight-lane transcript marker; backward-safe (Full-lane behavior unchanged when the marker is absent).
- New fixtures: `15_stop_allows_lightweight_deploy_complete` (allowed without Full-lane signals) and `16_stop_blocks_lightweight_deploy_complete_without_rollback` (still blocked when the safety floor is dropped). Tests **16/16**.

## Using the Lightweight Lane

Tell the PO about a small change as usual. If it passes the gate, the PO writes a one-line change-note and hands the AI Developer a single pass; you approve deploy with a plain "ship it." If it turns out non-trivial, the AI Developer stops and promotes to the Full lane. When in doubt, it's Full.
