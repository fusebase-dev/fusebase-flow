# Fusebase Flow v4.2.2 — third-reviewer findings + MAX_PATH manifest hardening

**Released:** 2026-07-09
**Type:** patch (follow-ups to v4.2.1; no new capabilities)

After v4.2.1 shipped, an independent **Opus 4.8 at maximum reasoning effort** confirmation pass (a third reviewer, after Codex `gpt-5.6-sol` xhigh and Fable 5) re-ran the release's repros and found it functionally sound — but caught two things both prior reviewers missed. This patch closes them, and lands Fable 5's previously-deferred manifest fix.

## Fixed

- **Completed the lock-vocabulary sweep.** v4.2.1's release notes claimed "no fictional `Locked: yes`", but the sweep that backed that claim used a `*.md`-only grep and missed two non-`.md` surfaces still teaching the removed marker: `.cursor/rules/fusebase-flow-specs.mdc` (a hand-condensed mirror of `communication`'s Mode-B principles — not covered by `mirror-skills --check`) and `docs/specs/delegation-resilience/spec.md`. Both now use the real `Lock status: LOCKED` convention; a repo-wide grep across **all** file types confirms zero real occurrences remain.
- **MAX_PATH manifest coverage (`hooks/local/lib/hook_manifest.py`).** On Windows with long paths disabled, `collect_assets()` silently omitted covered files whose absolute path exceeded ~260 chars: `pathlib.is_file()` swallows the underlying `WinError 3` to `False`, so an over-limit covered file dropped out of the manifest **at stamp time** with no error — a latent coverage/tamper-detection hole. (End-user `verify` already fails *closed*: it iterates the recorded paths, so an over-limit file reports `missing` → loud DRIFT, never a silent pass. CI stamps on shallow Linux paths, so the committed manifest could not silently shrink.) Fixed at the root: a `_resolve_root()` helper returns the Windows extended-length (`\\?\`) form, applied at all four stamp/verify entry points (`collect_assets`, `build_manifest`, `stamp`, `verify`). Verified: byte-identical manifest at normal depths (D1 byte-stability preserved), and a >260-char path is now stat/hashable where the plain path fails.

## Known / tracked (design-first spec planned — NOT fixed here)

The **FR-25 upgrade-adoption gap**, surfaced by a consumer project after its v4.2.0 upgrade:
1. The upgrade enables the FR-25 module-size ratchet and ships a **non-empty** `policies/module-size-baseline.txt` (Flow's own two over-ceiling files). The `check-module-size.sh` "warn-only *while no baseline*" adoption grace therefore never fires for a consumer with pre-existing over-ceiling files, so the first commit touching a consumer monolith hard-blocks with no smooth adoption.
2. The FR-25 remedy ("re-key the baseline") targets `policies/module-size-baseline.txt`, which is **FR-07-protected** — the sanctioned `--write-baseline` collides with a second guardrail, and the block message doesn't flag the FR-07 prerequisite.

This changes the interaction of two guardrails (FR-25 × FR-07) and touches protected internals, so it is being treated design-first rather than patched in haste. **Consumer workaround now:** operator-run `bash hooks/local/check-module-size.sh --write-baseline` (the intended "adopt FR-25 on an existing repo" step) and commit the baseline via a bootstrap approval (`write-bootstrap-approval.sh` mint → commit → `--consume`) — the FR-07-sanctioned path, not `--no-verify`.
