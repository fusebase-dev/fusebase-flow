# Fusebase Flow v3.8.4 — non-authoritative provider mirrors are benign, not drift (Issue 2)

**Release date:** 2026-06-01
**Previous version:** 3.8.3

## Summary

A downstream hit a false `CLI_LAYER_DRIFT` for CLI provider skills missing from `.agents/skills/` (e.g. `app-backend`, `app-routing`, `app-secrets`, `app-sidecar`), with the dead-end remediation "run `fusebase update`." Verified against the FuseBase CLI source: `fusebase update` writes CLI provider skills/agents to **`.claude/` only**. So the `.agents/`/`.codex/` provider mirrors are maintained by neither the CLI nor Flow (Flow never writes CLI provider skill text), and flagging their absence as drift — then advising a command that won't touch them — was wrong. v3.8.4 fixes it.

## Fixed

### Issue 2 — `.claude/` is the authoritative CLI-provider surface; `.agents/`/`.codex/` are benign mirrors

`check-cli-flow-conflicts.sh`:

- **Authoritative** (`.claude/skills`, `.claude/agents`): unchanged — full F4/U10 drift logic. A genuinely missing CLI provider skill here is still `CLI_LAYER_DRIFT` with the correct "run `fusebase update`" advice (which *does* write `.claude/`).
- **Non-authoritative** (`.agents/skills`, `.codex/agents`): now a single **benign INFO** — `"N/M present, K absent — absence is expected (the CLI maintains CLI provider skills in .claude only; Flow never writes CLI provider skill text). Copy from .claude/ if you want Codex/.agents parity."` Never `MISSING`/`CLI_LAYER_DRIFT`.

Net: a project with CLI provider skills present in `.claude/` but partially/wholly absent from `.agents/` now reads **HEALTHY**, and the only remediation shown is the accurate one (copy from `.claude/`, or accept the gap) — no dead-end `fusebase update`.

### `feature-*` vs `app-*` orphans — no Flow change needed

The CLI renamed `feature-*` provider skills to `app-*` and may leave orphans in `.claude/skills/`. Flow's health check only looks for the current `app-*` `known_names`, so orphaned `feature-*` dirs are invisible to it — no churn. (Cleaning them is the CLI's job.)

## Why this is correct (grounded in the CLI source)

`lib/copy-template.ts` / `lib/commands/product.ts`: `fusebase update`'s own summary is "Updated AGENTS.md, `.claude/skills`, `.claude/agents`, `.claude/hooks` and `.claude/settings.json`." It never writes `.agents/` or `.codex/` skills/agents. The CLI is moving to a `.claude/`-only skills model. So `.claude/` is unambiguously the authoritative CLI-provider surface; the other mirrors are best-effort and their absence is expected.

## Issue 1 (root `./skills` deprecation) — status

The v3.8.3 guard stands (health flags a deleted `skills/`; AGENTS overlay + README say ignore the CLI's "obsolete ./skills" warning while Flow is installed). The CLI source confirms the deprecation is real and directional — which **rules out** option #3 (mirror into `.claude/skills/` as Flow's source-of-truth), since the CLI owns and rewrites `.claude/skills/` on every update and would clobber Flow's canonical skills. The remaining choice is **keep root `skills/` (guarded, status quo)** vs **move Flow's canonical store to a Flow-namespaced path the CLI ignores** — an open operator decision; the guard makes "keep" safe with no deadline.
