# Fusebase Flow v2.4.0 — health-check deferral artifacts

**Released:** 2026-05-10

## What's new

A new operator-authored mechanism (`state/approvals/health_check_deferral-*.json`) for marking specific health-check drift items as deliberate-by-design rather than actual drift. Closes the architectural gap surfaced by `paperclip+hermes-v1`'s install brief: an install can deliberately omit parts of the canonical Fusebase Flow setup, and the engine will now correctly report `EXCEPTION_IN_EFFECT` instead of `DRIFTED` / `BROKEN`.

## The gap v2.4.0 closes

Pre-v2.4.0, the engine had only one form of drift attribution — the `protected_path_edit-*.json` artifact mechanism, which only applied to specific hook test failures. There was no way to tell the engine "this specific check is deliberately deferred."

Real-world consequence: `paperclip+hermes-v1`'s install brief explicitly deferred two checks per Steps 9 + 10 of its install discipline:

| Brief step | Deferred check | Why |
|---|---|---|
| Step 9 | `.claude/settings.json` lifecycle hooks NOT wired | Preserve project's pre-existing quality-check + lint-on-stop hooks |
| Step 10 | Windows `shell:true` patch NOT applied to `.claude/hooks/run-typecheck-features.js` | Step 10 listed `.claude/hooks/` as protected per the project's discipline |

Brief's Step 15 expected `HEALTHY` after install. The engine reported `BROKEN` instead, because it had no concept of "approved deferral" for these check types. Brief's expectation was correct — the engine was the gap.

v2.4.0 adds the missing primitive.

## How it works

### 1. Operator authors a deferral artifact

```json
{
  "approved_by": "operator@example.com",
  "scope": "Lifecycle hooks + Windows patch deferred per install brief 2026-05-08",
  "expires_at": "2026-08-10T00:00:00Z",
  "reason": "Project preserves existing hooks per Step 9; .claude/hooks/ protected per Step 10",
  "deferred_checks": [
    "settings_json_lifecycle_events",
    "windows_shell_patch"
  ]
}
```

Filename: `state/approvals/health_check_deferral-paperclip-install-discipline-20260510.json`

### 2. Engine processes the artifact

On startup (Section 0), the engine:

- Scans `state/approvals/*.json` for files matching `health_check_deferral-*.json`
- Filters out expired artifacts (per `expires_at`)
- Aggregates each artifact's `deferred_checks` array into a global `DEFERRED_CHECKS` list
- Tracks the originating artifact filename in parallel (for output traceability)

### 3. Each defer-able check site uses `record_drift`

Instead of pushing directly to `LOCAL_DRIFT`, defer-able checks call `record_drift <check_id> <message>`. The helper looks up `<check_id>` in `DEFERRED_CHECKS`:

- **Found:** message goes to `LOCAL_DEFERRED` with a `[check_id=<id>; deferred per <artifact>]` suffix
- **Not found:** message goes to `LOCAL_DRIFT` as before

### 4. Verdict logic

```
LOCAL_BROKEN non-empty → BROKEN (deferrals can't override real failures)
LOCAL_DRIFT empty AND LOCAL_DEFERRED non-empty → EXCEPTION_IN_EFFECT (exit 3)
LOCAL_DRIFT non-empty → DRIFTED or signature-specific verdicts (existing logic)
Everything OK → HEALTHY
```

### 5. Output

```
Local state (12 checks):
  ✓ VERSION file: 2.4.0
  ✓ AGENTS.md overlay block: present
  ✓ CLAUDE.md overlay block: present
  ✓ .claude/skills/: 10/10 mirrored
  ✓ .claude/agents/: 2/2 mirrored
  ✓ ...
  ⊘ .claude/settings.json: only 1/6 lifecycle events wired [check_id=settings_json_lifecycle_events; deferred per health_check_deferral-paperclip-install-discipline-20260510.json]
  ⊘ Windows shell:true patch: MISSING [check_id=windows_shell_patch; deferred per health_check_deferral-paperclip-install-discipline-20260510.json]

Deferred checks (2 — operator-authored exceptions; engine v2.4.0+):
  Each ⊘ above is reclassified from drift to deferred via a
  state/approvals/health_check_deferral-*.json artifact. ...

Active approval artifacts (1):
  • health_check_deferral-paperclip-install-discipline-20260510.json: deferred_checks=2 expires=2026-08-10T00:00:00Z scope="..."

Verdict: EXCEPTION_IN_EFFECT
```

## Canonical defer-able check_id taxonomy

6 check_ids the engine recognizes:

| `check_id` | What it defers |
|---|---|
| `agents_md_overlay` | AGENTS.md overlay block missing |
| `claude_md_overlay` | CLAUDE.md overlay block missing |
| `settings_json_lifecycle_events` | `.claude/settings.json` lifecycle events count below upstream-canonical |
| `claude_skills_mirror_count` | `.claude/skills/` mirror count below upstream-canonical |
| `claude_agents_mirror_count` | `.claude/agents/` mirror count below upstream-canonical |
| `windows_shell_patch` | Windows `shell:true` patch on run-typecheck-features.js not applied |

Critical-infrastructure checks (preflight failures, recovery script presence, overlay templates folder, hook test failures other than `protected_path_edit*`, the health-check skill's own self-presence, AGENTS.md/CLAUDE.md DUPLICATE blocks) are deliberately NOT defer-able. Engine bypasses the deferral mechanism for these.

Full reference: [`docs/health-check-deferrals.md`](../health-check-deferrals.md).

## Latent v2.2.1 bug fixed in passing

Surfaced during v2.4.0 development: the AGENTS.md / CLAUDE.md overlay-marker count check used `grep -cF ... || echo 0` which produced corrupted `"0\n0"` output when count was 0 (same `set -o pipefail` interaction as v2.3.0's diff-count bug). Fixed by replacing `|| echo 0` with `|| true` — same pattern as v2.3.1's fix. Latent since v2.2.1's introduction of duplicate-marker detection; only triggered when a project genuinely lacked the overlay marker.

## What this enables

| Before v2.4.0 | After v2.4.0 |
|---|---|
| Install brief's "Step 15: expect HEALTHY" was a wrong prediction when Steps 9-10 deferred parts of the install | Brief can predict HEALTHY-given-deferral and the engine agrees (returns EXCEPTION_IN_EFFECT) |
| Operators with deliberate scope reductions had to either (a) ignore BROKEN noise, (b) document workarounds in AGENTS.md, or (c) modify the engine locally | Operators file an artifact; engine respects the deferral; verdict reflects reality |
| `paperclip+hermes-v1` AGENTS.md doc note (added in commit `ad6127d`) was the workaround | The doc note remains useful as human-readable context, but the artifact is the canonical decision record |

## Notes for upgraders (v2.3.2 → v2.4.0)

- **Purely additive feature.** No content changes; no migration needed for projects that don't author deferral artifacts.
- Upgrade path: `cd .fusebase-flow-source && git pull origin main && cd .. && bash hooks/local/upgrade-engine.sh` — script self-update picks up v2.4.0.
- Existing `protected_path_edit-*.json` artifacts continue to work unchanged.
- Read `docs/health-check-deferrals.md` if you have install briefs that deliberately omit parts of canonical setup.

## What didn't change

- Recovery script (`hooks/local/post-fusebase-update.sh`) — unchanged
- All overlay templates — unchanged
- All 10 skills + 2 sub-agents — unchanged
- Slash command — unchanged
- Settings merger — unchanged
- `upgrade-engine.sh` — unchanged from v2.3.2

This is a feature add scoped tightly to the engine + a new artifact type. No surface area beyond that.
