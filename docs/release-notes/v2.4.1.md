# Fusebase Flow v2.4.1 — Windows CRLF guard for deferral mechanism

**Released:** 2026-05-10

## TL;DR

Patch release. Fixes a Windows-specific bug in v2.4.0's deferral mechanism where multi-entry `deferred_checks` lists silently failed to match — every entry except the last gained a stray `\r` from Python-to-bash output handling, so `record_drift` never recognized them as deferred.

Single-entry deferral artifacts and all Linux/macOS use of v2.4.0 were unaffected.

## Who needs to upgrade

| Environment | Action |
|---|---|
| Windows + multi-entry deferral artifact | **Upgrade required.** Without v2.4.1 the engine misclassifies drift the operator has explicitly authorized. |
| Windows + single-entry deferral artifact | Upgrade recommended. No live bug today, but any future addition to `deferred_checks` triggers it. |
| Linux/macOS, any deferral configuration | Optional. The Python `print()` boundary doesn't emit `CRLF` on POSIX, so v2.4.0 already worked correctly. The v2.4.1 strips are no-ops there. |
| No deferral artifacts at all | No upgrade needed for this fix; v2.4.0 remains correct for non-deferral flows. |

`paperclip+hermes-v1` operators: see "Coordination with downstream" below.

## Root cause

The engine reads `deferred_checks` from JSON artifacts via a Python helper, then iterates lines in bash:

```bash
deferred_list=$(python3 - "$artifact_file" <<'PY'
# ... prints one check_id per line ...
PY
)
while IFS= read -r cid; do
  DEFERRED_CHECKS+=("$cid")
done <<< "$deferred_list"
```

Two interacting platform behaviors:

1. **Python `print()` on Windows emits `CRLF`** (`\r\n`) regardless of `PYTHONIOENCODING`. The line-ending strategy is set by `sys.stdout`'s text-mode buffer, not by the encoding.
2. **Bash `read -r` strips trailing `LF` but not `CR`**. Combined with `$()` command substitution (which only strips trailing `LF` from the captured stdout, not embedded `CR` characters between lines), every entry in a multi-line list ends up with a trailing `\r` — except the last one, where `$()` happens to strip the final `\r\n` cleanly.

Concrete example (Windows, `deferred_checks: ["agents_md_overlay","claude_md_overlay","claude_skills_mirror_count"]`):

```
DEFERRED_CHECKS=("agents_md_overlay\r" "claude_md_overlay\r" "claude_skills_mirror_count")
```

Then `record_drift` compares against unadorned strings like `"agents_md_overlay"`, finds no match for the first two, and pushes them to `LOCAL_DRIFT` instead of `LOCAL_DEFERRED`.

The bug was masked in v2.4.0's smoke test because:

- Smoke test ran on Linux (no `CRLF` from Python).
- Single-entry deferrals on Windows happened to work — the lone entry has no `\r` suffix.

The receiving agent on `paperclip+hermes-v1` caught it within hours of v2.4.0 landing, because the install brief defers exactly two checks. They diagnosed it via `od -c` showing `\r` between entries and reported it back.

## The fix

Defensive `\r` strip applied at all three Python-to-bash boundaries in `hooks/local/fusebase-flow-health-check.sh`:

### 1. `deferred_checks` while-read loop (the original bug site)

```bash
while IFS= read -r cid; do
  cid="${cid%$'\r'}"          # ← v2.4.1
  [ -z "$cid" ] && continue
  DEFERRED_CHECKS+=("$cid")
done <<< "$deferred_list"
```

### 2. `EXPECTED_EVENTS_STR` events parser (parallel boundary, theoretical bug)

```bash
EXPECTED_EVENTS_STR="${EXPECTED_EVENTS_STR//$'\r'/}"   # ← v2.4.1
for e in $EXPECTED_EVENTS_STR; do EXPECTED_EVENTS+=("$e"); done
```

The events string is whitespace-split, so a trailing `\r` would attach to the last event name (e.g. `PreCompact\r`), causing `grep -q "\"PreCompact\":"` to miss. Same class of bug; defensive fix.

### 3. `summary` capture (cosmetic)

```bash
summary="${summary//$'\r'/}"   # ← v2.4.1
```

Would have only caused a trailing `\r` in `ARTIFACT_NOTES` console output, not a logic bug. Included so all three boundaries are uniformly defended.

All three strips are idempotent on Linux/macOS (no `\r` characters to remove).

## Verification

Smoke test in a fresh project on Windows with a multi-entry deferral artifact:

```json
{
  "approved_by": "smoke-test@local",
  "scope": "v2.4.1 multi-entry CRLF regression test",
  "expires_at": "2026-12-31T00:00:00Z",
  "reason": "Verify all three entries classify as LOCAL_DEFERRED",
  "deferred_checks": [
    "agents_md_overlay",
    "claude_md_overlay",
    "claude_skills_mirror_count"
  ]
}
```

**Pre-v2.4.1 (v2.4.0 engine):** `agents_md_overlay` and `claude_md_overlay` reported as `LOCAL_DRIFT`; only the last entry classified as `LOCAL_DEFERRED`. Verdict: `DRIFTED`.

**v2.4.1 engine:** All three classify as `LOCAL_DEFERRED`. Verdict: `EXCEPTION_IN_EFFECT` (exit code 3).

## Upgrade

Same path as any v2.x → v2.x upgrade:

```bash
bash hooks/local/upgrade-engine.sh
```

The engine self-update is one-file (`hooks/local/fusebase-flow-health-check.sh`); the upgrade script will detect the SHA difference and apply the new version. No other engine files change in this release.

## Coordination with downstream

`paperclip+hermes-v1` carries an equivalent fix as a local engine patch on its branch (committed against upstream v2.4.0 with deviation note). Operators upgrading `paperclip+hermes-v1` to v2.4.1 via `upgrade-engine.sh` can drop the local patch — upstream and downstream now converge on the same engine bytes.

The local patch and v2.4.1 are byte-equivalent on the deferred-checks loop; the upstream version additionally strips the events-parser and summary-capture boundaries (defensive; no live bug).

## Backward compatibility

v2.4.1 is a strict patch over v2.4.0:

- All v2.4.0 deferral artifacts continue to work unchanged.
- All check_ids, verdict logic, and exit codes are identical to v2.4.0.
- No new operator surface, no new artifact schema, no new check_ids.
