# Fusebase Flow v3.8.2 — flag-gated CLI skills are absent-by-design, not drift (U10)

**Release date:** 2026-06-01
**Previous version:** 3.8.1

## Summary

A downstream reported a **chronic false-positive `CLI_LAYER_DRIFT`**: the health check flagged `fusebase-portal-specific-apps` and `managed-integrations` as `MISSING`, but they're **flag-gated** — the FuseBase CLI deletes them when their config flag is off, so they're absent by design. The advised remediation (`fusebase update`) can never restore a flag-off skill. This is the same class as F4 (absent-by-design ≠ drift) and affected nearly every downstream, since few enable every optional flag. Fixed in v3.8.2.

## Fixed

### U10 — health check is now flag-gating aware

The FuseBase CLI gates several provider skills behind config flags (`lib/copy-template.ts` `FLAG_GATED_SKILLS`) and removes the skill dir when the flag is off:

| Skill | Enabling flag |
|---|---|
| `git-workflow` | `git-init` / `git-debug-commits` |
| `app-business-docs` | `app-business-docs` |
| `mcp-gate-debug` | `mcp-gate-debug` |
| `fusebase-portal-specific-apps` | `portal-specific-apps` |
| `managed-integrations` | `managed-integrations` |

- `hooks/local/fusebase-flow-overlays/agent-surface-ownership.json` gains a `flag_gated_skills` map mirroring that list.
- `hooks/local/check-cli-flow-conflicts.sh` resolves each flag-gated skill:
  - **flag off / undeterminable + absent → benign INFO** (not drift), naming the correct remediation: `fusebase config set-flag <flag>` — *only if you want the feature*.
  - **flag provably on + absent → genuine `CLI_LAYER_DRIFT`** (best-effort read-only resolution from `fusebase.json`; the diagnostic never shells out to the `fusebase` CLI).
  - present → OK (unchanged).
- **Non-flag-gated** provider skills are unaffected — a genuinely missing one (e.g. `fusebase-cli`) still escalates to `CLI_LAYER_DRIFT`. Precision retained, proven by the existing test.

Result: a downstream that simply didn't opt into `portal-specific-apps` / `managed-integrations` now reads **HEALTHY** instead of chronic `CLI_LAYER_DRIFT` with dead-end advice.

## Note for downstreams

If your health check showed `CLI_LAYER_DRIFT` only for flag-gated skills you intentionally didn't enable, that verdict is now `HEALTHY`. If you *want* one of those features, enable it with `fusebase config set-flag <flag>` and re-run the CLI — `fusebase update` alone won't add a flag-off skill.
