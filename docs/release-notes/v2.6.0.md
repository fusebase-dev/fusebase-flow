# Fusebase Flow v2.6.0 — Operator Stewardship

**Released:** 2026-05-10

## TL;DR

This release codifies a single design principle as a framework rule:

> **The human operator should only do what only humans can do** — make product decisions, approve gates, and physically move messages between sessions. Everything else (interpreting reports, recommending next steps, composing prompts to paste back) is the agent's job.

Three concrete additions enforce this:

1. **FR-16** — new always-on rule in `FLOW_RULES.md`. Self-attestation language updated to "FR-01 through FR-16."
2. **Operator Relay Protocol** — mandatory 5-step ritual added to PO's role-discipline. When the operator pastes output from another role, PO must analyze → brief in Mode A → recommend with #1 marked → await approval → generate verbatim paste-back prompt.
3. **Return-path templates** — `templates/gate-report.md`, `templates/deploy-report.md`, `templates/architect-response.md`. Each template ends with an operator-relay block the AI Developer / Deploy / Architect must fill; operator copies that block into PO chat instead of scanning the technical body.

No engine changes. No `hooks/local/` script changes. Pure framework + templates + role-discipline work.

## What you'll notice as an operator

Same scenario you may have hit before: AI Developer hits the gate and produces a 50-line technical report. Pre-v2.6.0, you'd scroll the report looking for "what's the headline / what should I tell PO?" and PO would respond to your question with framework jargon.

Post-v2.6.0:

| Step | What you do | What v2.6.0 changes |
|---|---|---|
| AI Developer reports gate | You receive the report | Report now ends with a clearly-marked `## For operator: paste this in PO chat` block |
| You relay to PO | Copy that block (not the whole report) → paste in PO chat | The block is ~10 lines of plain English, not 50 lines of Mode B technical content |
| PO briefs you | Reads the block + the full report | PO must follow the **Operator Relay Protocol**: 4-sentence brief in Mode A, options table with #1 marked, **no framework jargon** |
| You decide | Approve recommended option (or override) | PO waits for explicit approval before composing the paste-back |
| PO gives you next prompt | The verbatim text to paste in AI Developer / Deploy chat | No more "what should I send?" — the prompt is fully composed, code-fenced, ready to copy |

The cumulative effect: round-trips drop from ~4 to ~2 for the same operator decision. Cognitive load on operator drops to **decision-making + button-pressing**.

## Backward compatibility

v2.6.0 is a strict superset of v2.5.0:

- Existing handoffs continue to work
- Existing reports without operator-relay blocks continue to work (operators just paste the whole report; PO does extra work to extract the headline)
- Existing PO sessions that don't follow the Operator Relay Protocol still function (PO.10 is enforced via role-discipline, not via hook — drift is detectable but not blocked)
- All v2.5.0+ deferral artifacts continue to work
- All v2.4.0+ check_ids, verdict logic, exit codes — identical

The new behavior surfaces gradually: as operators upgrade and use new handoffs / new reports, the relay structure becomes default.

## Upgrade

Same path as any v2.x → v2.x upgrade:

```bash
cd .fusebase-flow-source && git pull origin main && cd ..
bash hooks/local/upgrade-engine.sh
```

For projects that maintain `AGENTS.md`/`CLAUDE.md` overlay blocks (most projects), run the recovery script after the engine sync to refresh overlay blocks with the new FR-16 reference:

```bash
bash hooks/local/post-fusebase-update.sh
```

The recovery script is additive + idempotent. It will detect the overlay-template changes and re-append from the updated overlay templates.

## Operator stewardship — design thesis

The framework's design has always been a balance: enforce discipline (FR rules) without imposing cognitive load on the operator. v2.0 through v2.5 mostly enforced discipline at the agent boundary — between sessions. v2.6 extends the discipline to the **operator-as-relay** boundary: the framework now declares that the operator's job is narrow and protects them from the cognitive overflow that crosses session boundaries.

The technical mechanism is simple: structure the artifacts (return-path templates) so the relay-block exists by default, and codify the protocol (Operator Relay Protocol) so PO behavior is consistent across sessions. The philosophical mechanism is the same: explicit-by-design. FR-16 is now part of every session's self-attestation; sessions that don't honor it are visibly drifting.

If the operator finds themselves composing prompts to paste back, parsing framework jargon to figure out next steps, or scrolling reports to find headlines — that's the framework drifting on FR-16. Surface it; the framework should be doing better.
