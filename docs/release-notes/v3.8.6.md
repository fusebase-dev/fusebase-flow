# Fusebase Flow v3.8.6 — staging clone no longer breaks deploy lint, + install UX

**Release date:** 2026-06-01
**Previous version:** 3.8.5

## Summary

From a live overlay project (Vite/React/TS, ESLint flat config, deploy via `fusebase deploy`): one **deploy blocker** and two minor install-UX items.

## Fixed

### [BLOCKER] `.fusebase-flow-source/` failed the project's ESLint → `fusebase deploy` failed

`fusebase deploy` runs lint first. Flow stages the upstream clone at `.fusebase-flow-source/`, which contains **CLI-owned CommonJS** hooks (`.claude/hooks/*.js` using `require()`). Those trip `@typescript-eslint/no-require-imports`. The path is gitignored, but **ESLint flat config does not read `.gitignore`**, and the CLI's `eslint.config.mjs` only ignores `.claude/**` — so the staged clone got linted and `npm run lint` exited 1 even with zero application errors.

Why the obvious options don't apply: Flow has **no eslint config of its own** (it doesn't inject `.claude/**` — the CLI does), and the failing hooks are **CLI-owned** (`audit/cli-vendor-manifest.json`) so Flow rewriting them to ESM would diverge from the CLI and be re-clobbered on `fusebase update`. So the fix keeps the staging clone from being linted:

- **New `hooks/local/eslint-ignore-flow-paths.sh`** (opt-in, idempotent, backs up): adds `".fusebase-flow-source/**"` to the project's ESLint flat-config `ignores`, placed right after `".claude/**"` (matching quote/comma style). No-op if already present or no flat config; prints guidance if no `.claude/**` anchor is found.
- **`upgrade.sh` / `bootstrap-upgrade.sh`** now print a loud closing note: the clone is **transient** — `rm -rf .fusebase-flow-source` after an upgrade (re-created on the next one), **or** run the helper.
- **Docs:** AGENTS-overlay "Maintenance posture" + README upgrade section document it.

Verified on the real CLI `project-template/eslint.config.mjs` shape: helper adds the entry, keeps the array valid (`node --check` passes), and is idempotent. Regression test **U15**.

### [MINOR] Project-values placeholders point at `/onboard`

The `### Project-specific values` table read "(customize during install)" with no cue. It now reads "(run `/onboard` or edit)" plus a note that `/onboard` is the canonical fill step and values are preserved across overlay refreshes (the U1 `FLOW:PRESERVE` region).

### [MINOR] Cold-start docs layout is now discoverable

README documents that `docs/specs/`, `docs/handoff/`, `docs/changes/`, `docs/backlog/` are created on demand (nothing to scaffold), so the expected layout is clear before the first PO session. (No empty `.gitkeep` clutter is shipped — the tree stays clean until work needs it.)

## Note for the reporting project

Re-pull 3.8.6, then either delete `.fusebase-flow-source/` or run `bash hooks/local/eslint-ignore-flow-paths.sh` once — your local `eslint.config.mjs` patch is then maintained by Flow's helper and won't silently drift on the next `fusebase update`. `fusebase deploy` lint will pass with the staging clone present.
