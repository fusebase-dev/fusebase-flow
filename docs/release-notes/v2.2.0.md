# Fusebase Flow v2.2.0 — Health check & recovery

**Released:** 2026-05-10

## What's new

A built-in **health check skill** + **recovery script** that diagnose and repair Fusebase Flow overlay drift. The most common drift cause is `fusebase update` (Fusebase CLI), which regenerates `AGENTS.md`, `.claude/settings.json`, and `.claude/hooks/run-typecheck-features.js` from CLI templates and evicts the Fusebase Flow overlay. v2.2.0 handles this end-to-end.

## Quick start

In any Fusebase Flow v2.2+ project:

```bash
# Diagnose (read-only)
bash hooks/local/fusebase-flow-health-check.sh

# Or in Claude Code:
/fusebase-health

# Or ask any AI agent:
"Is Fusebase Flow healthy?"
```

If drift is detected, the skill **offers** to run recovery in chat. Reply `yes` (or `run it` / `fix it` / `proceed`) to execute `bash hooks/local/post-fusebase-update.sh` automatically + re-check.

## Five verdicts

The engine emits one of five verdicts, each with a distinct exit code:

| Verdict | Exit | Meaning | Recommended action |
|---|:---:|---|---|
| `HEALTHY` | 0 | All checks pass; upstream in sync | None |
| `EXCEPTION_IN_EFFECT` | 3 | All drift attributable to active approval artifacts in `state/approvals/` | Review/clear the artifact (NOT recovery — recovery doesn't touch state/approvals/) |
| `FUSEBASE_UPDATE_AFTERMATH` | 1 | Canonical `fusebase update` aftermath: AGENTS.md overlay missing AND `.claude/settings.json` reduced | Recovery offered in chat |
| `DRIFTED` | 1 | Drift detected but doesn't match a known pattern | Manual investigation; recovery as optional fallback (still confirmed) |
| `BROKEN` | 2 | Genuine failure NOT attributable to operator-authored exceptions (e.g. genuine hook test failure, missing recovery infrastructure) | Manually inspect LOCAL_BROKEN items |

## The diagnose-then-offer pattern

The skill is **read-only during diagnosis**. When drift is detected and recoverable, the skill offers recovery in chat:

```
Run recovery now? It will:
  • Restore AGENTS.md overlay block (if missing)
  • Merge .claude/settings.json lifecycle events (if reduced)
  • Re-apply Windows shell:true patch (if missing)
  • Re-mirror Fusebase Flow skills + sub-agents (no-op if already present)

Reply `yes` / `run it` / `fix it` / `proceed` to execute.
Reply anything else to halt and decide later.
```

On affirmative reply → recovery executes + re-check + report new verdict. On any non-affirmative reply (silence, `no`, a question) → halt. **No nagging.**

This preserves PO.5 (operator decides) while reducing friction — no terminal context-switch needed for the common case.

## Auto-discovery for upstream upgrades

The engine and the merger auto-discover canonical sets at runtime from `.fusebase-flow-source/`:

- **Skill names** from `skills/*/`
- **Agent names** from `agents/*/`
- **Lifecycle event names** from `.claude/settings.json.example`
- **Hook handler commands + matchers** from the same example file

**Patch / minor upstream releases** (new skill / agent / event) require **zero maintenance** to this system. Only major-version semantic changes (e.g. heading marker rename) require manual edits.

## What the recovery script restores

10 idempotent steps. Each checks state first, applies only if missing.

1. Re-mirror Fusebase Flow skills (`bash hooks/local/mirror-skills.sh` — preserves any operator-added skills).
2. Re-mirror Fusebase Flow sub-agents (`bash hooks/local/mirror-agents.sh`).
3. Re-append `## Fusebase Flow — workflow lifecycle overlay` block to `AGENTS.md`.
4. Re-append `## Fusebase Flow — additional rules (overlay)` block to `CLAUDE.md`.
5. Re-merge `.claude/settings.json` to add missing lifecycle event keys + append Fusebase Flow `stop.py` to the Stop chain (via the Python merger; auto-discovers from upstream).
6. Re-apply Windows `shell:true` patch on `.claude/hooks/run-typecheck-features.js` (CVE-2024-27980 mitigation; only on Windows).
7. Reserved.
8. Reserved.
9. Restore `fusebase-flow-health-check` skill mirror (if missing or out of sync).
10. Restore `/fusebase-health` slash command at `.claude/commands/fusebase-health.md` (if missing or out of sync).

Idempotency: running the script a second time produces "already in place" for every restorable item.

## Heading marker convention

This release standardizes on:

- `## Fusebase Flow — workflow lifecycle overlay` (AGENTS.md)
- `## Fusebase Flow — additional rules (overlay)` (CLAUDE.md)

The previous v2.1.x convention used "Fusebase Flow V2 — workflow lifecycle overlay" (with a "V2" qualifier). v2.2.0 drops the "V2" to align with the standard "Fusebase Flow" naming.

**Upgraders from v2.1.x:** see CHANGELOG.md → "Notes for upgraders". TL;DR — manually edit your AGENTS.md and CLAUDE.md headings to drop the "V2 ", or your installation will look like drift to the new health check.

## Why this matters

Before v2.2:

- `fusebase update` could silently degrade your Fusebase Flow overlay. Operators discovered the breakage hours later when their agent stopped following Mode A/B, role-discipline, or running the lifecycle hooks.
- Recovery required either manual edits or remembering exact bash commands.
- No way to verify "is my overlay still intact?" without manually grepping AGENTS.md.

After v2.2:

- `/fusebase-health` (or natural-language equivalent) gives a 5-second tabular diagnosis.
- One-word in-chat reply (`yes`) executes idempotent recovery + re-checks.
- Engine + recovery auto-adapt to most upstream releases (no maintenance burden on minor versions).

## Operator authority preserved (PO.5)

The skill never runs recovery without explicit operator confirmation. Silence, `no`, a counter-question, or anything that isn't an unambiguous affirmative reply halts. The operator owns the trigger; the skill just makes the decision low-friction.
