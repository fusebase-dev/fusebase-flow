## Changed

### M (MED) — durable updater-written receipt closes the v3.30.0 silent-non-detection blind spot

v3.30.0 diff-framed `SHARED_MERGE_DRIFT` against the pre-merge backup `.claude/settings.json.pre-flow-merge`. But `post-fusebase-update.sh` writes that backup before merging and `rm -f`s it on the no-op ("already up to date") path — so after the steady state (merge once, then no-op updates) the backup is gone, and a genuinely-dropped CLI Stop hook fell through to `OK` **silently**, with no signal that verification was skipped.

- **Writer (`settings-json-merge.py --baseline-out` via `post-fusebase-update.sh --wire-hooks`).** The merge now writes the CLI-owned Stop commands it preserved to a durable receipt `state/audit/cli-stop-baseline.json` (a command is CLI-owned iff it names a file under `.claude/hooks/` — the writer single-sources the list the reporter reads; `stop.py` is excluded). It runs on **every** wire-hooks invocation — after a real merge AND on the no-op path — so the receipt is durable and **self-refreshing** (re-running the updater re-baselines = the clear/escape path). The ephemeral `.pre-flow-merge` is no longer the diff source (it stays only the merge-failure restore source); there is **never** an on-disk-`.claude/hooks/` fallback (that would re-introduce the v3.30.0 `run-typecheck-apps.js` false positive, which 0.25.9 ships unwired).

### Advisory-only reclassification — the v3.30.0 exit-1 path is REMOVED

The reporter now reads the receipt and emits only advisory findings:

- `has_flow_stop` true + **no receipt** → advisory **`CLI_STOP_UNVERIFIED`** ("cannot verify CLI Stop preservation; re-run `post-fusebase-update.sh --wire-hooks` to establish a baseline"). Never fires on a no-`stop.py` project.
- receipt present + a baselined CLI Stop command **missing now** → advisory **`CLI_STOP_BASELINE_DRIFT`** ("a CLI Stop hook wired at last Flow update is gone; re-run the updater to re-baseline if intentional").

**The v3.30.0 `SHARED_MERGE_DRIFT`/exit-1 path for a missing CLI Stop hook is REMOVED.** Rationale: Flow's merge is provably **preserve-only**, so a "missing" hook is never a Flow-merge fault (it's a non-Flow/operator edit) — classifying it exit-1 was a permanent false positive. A project carrying only these new findings stays **HEALTHY / exit 0**. The merge's preserve-only correctness remains guarded by the merge's own idempotency / older-CLI tests (unchanged) — removing the exit-1 path stops miscategorizing non-merge edits as a merge fault; it does not weaken merge verification.

### L (LOW) — `CLI_CUSTOM_AT_RISK` gated on provenance drift (sha-gate)

`scan_custom_skill_block` previously flagged ANY `CUSTOM:SKILL` block in a CLI-owned skill — including a **pristine CLI-shipped** block (e.g. `app-dev-practices`), which re-vendor simply overwrites with the identical CLI copy (nothing at risk). It now fires only when the file has **drifted from provenance** (sha256 ≠ bundled provenance = operator content a CLI refresh would clobber). sha == provenance → CLI-shipped → skip; provenance unavailable for that file → conservative flag (genuine signal preserved). Advisory-only contract unchanged.
