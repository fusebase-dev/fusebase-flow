# Fusebase Flow v4.4.0 - Codex Product Owner discovery + plugin packaging; natural deploy-approval phrase

**Released:** 2026-07-11
**Type:** minor (Codex distribution and command-discovery UX; deploy-approval operator UX)

## Why

Codex does not currently offer the same native bare slash-command surface as Claude Code. In Codex, Flow commands are best exposed as plugin skills, but Product Owner activation was still only documented as a portable command/agent route. Searching `/product` in Codex therefore showed long, noisy Flow-prefixed skill rows and missed the most important activation target: Product Owner.

## What ships

1. **Codex plugin manifest.** `.codex-plugin/plugin.json` declares the Flow plugin, points Codex at `./.agents/skills/`, and uses the shorter display name `Flow` to reduce the provenance text Codex may render in skill search rows.
2. **Product Owner skill bridge.** New canonical `flow-skills/product-owner/SKILL.md` routes `/product-owner`, Product Owner activation requests, and Codex `/product` searches to the existing Product Owner role/agent contract without duplicating the role body.
3. **Provider mirrors.** `mirror-skills.sh` now emits the Product Owner skill into both `.agents/skills/product-owner/` and `.claude/skills/product-owner/`; manifests and docs now record 33 canonical Flow skills.
4. **Regression coverage.** New `hooks/tests/test-codex-plugin-surface.sh` verifies the plugin manifest, Product Owner bridge, mirrored copies, and trigger text. Existing Codex prompt parity still passes.
5. **Release surface updates.** Install, compatibility, publishing, upgrade, partial-upgrade, preflight, backlog, and CI allowlists now include `.codex-plugin/`.

## Deploy-approval phrase — natural & forgiving (operator UX)

An operator reported that the one gate Flow deliberately keeps — the deploy-approval phrase — had become ceremonious: an agent asked them to type a long composed sentence (*"Publish M12-A #272 to production — auth/permission change approved. APPROVE-DEPLOY-NOW"*) plus "(or your words + the phrase)", bundled with three other questions. Nothing in the framework required that; the agent improvised it, and the framework wording was loose enough to allow it.

- **Forgiving phrase.** The Full-lane DP.6 phrase is now **`approve deploy now`** — matched forgivingly (any case / spacing / hyphenation; legacy `APPROVE-DEPLOY-NOW` still passes), but it must be that deliberate three-word phrase; a bare `yes`/`ok`/`go` still does not count (the ~5s attentiveness friction survives). Lightweight-lane (DP.12) keeps its plain go-ahead. This is a **behavioral gate only** — `approve-local.sh` takes the reason as free text and enforcement checks the artifact filename + `expires_at`, never the phrase — so no enforcement logic changed.
- **Presentation guardrail.** The operator types **only** the phrase. The agent must never make them compose a sentence, prepend the ticket/scope, type "your words + the phrase", or answer other questions in the same message — scope is the agent's own one-line summary shown above the ask. Reworded across `deploy.md`, the Operator Gate Protocol, `greenlight-deploy.md`, the `ai-developer` AGENT, `lightweight-lane`, the deploy templates, `approval-policy.yml`/`ratchet-governance.yml`, and the provider security rules.
- **Audit detector** (`find_wasted_effort/evidence.py`) made separator-agnostic with a bounded, word-boundaried regex (no false positives on `approvedeploynow` / `disapprove deploy now`); selftest extended with mixed-separator + negative cases.

Reviewed by Codex gpt-5.6-sol (High): 2 blockers + 1 medium (stale hook manifest, mid-edit mirror drift, regex over-match) found and fixed; both find-wasted-effort selftests pass, mirror `--check` 0-drift.
