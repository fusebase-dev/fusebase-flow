# Fusebase Flow v3.10.0 — FR-22 code-comment policy

**Release date:** 2026-06-04
**Previous version:** 3.9.0

## Summary

Adds **FR-22**, a new always-on rule: write only **tripwire** + **retrieval-pointer** comments, and remove everything else. In a Flow workflow source files are read by AI agents, not humans — so WHAT-restating prose, recorded-elsewhere rationale, and changelog history serve an absent audience and cost context budget on every load. Cross-project audits (paperclip+hermes-v1 + AssetWatch Prod, 2026-06-04) measured ~45% of comments removable in trust-critical files, concentrated in dense engine/auth files (CRUD is already lean).

## Why a framework rule

Two root causes are only fixable at the framework level:

1. The base harness instruction **"match surrounding comment density"** is a one-directional ratchet — density only ever increases. FR-22 ships the explicit override.
2. **No feedback penalty** — every Stop-hook gate (typecheck/lint) is comment-blind. Wrong code fails a gate; a useless comment fails nothing. So enforcement moves to write-time (FR-22) + review-time (`code-review`).

## The rule (FR-22)

Write only two kinds of comment; remove everything else:

- **Tripwire** — a constraint an editing agent could violate without realizing and that isn't obvious from local code (`empirical floor — don't lower below X`; `additive — editing breaks back-compat`; an auth/platform/concurrency quirk). One line by default; ≤~4 lines only for security/auth/concurrency/platform-quirk.
- **Retrieval pointer** — a ≤1-line tag naming the external WHY-home (`(decision B2)`, `backlog 156`).

Remove WHAT-restating comments, rationale already recorded in a decision/ticket/memory (→ pointer), and changelog/history (→ git). **Do not match surrounding density upward.** Carve-out: trust-critical paths keep multi-line tripwires.

## Two subtleties preserved (not over-simplified)

- **Storage ≠ retrieval → the pointer is NOT a duplicate.** When an agent opens a file, the decision/backlog records aren't in its context. The one-line pointer is the only in-context trigger to that external record; deleting it orphans the record. Kill the prose, keep the pointer.
- **Architecture-dependent → carve-outs are project-configurable.** Whether a separate instruction layer is read instead of source varies by project (SUPPORTED in one audited repo, REFUTED in another). The trust-critical set is project-settable in `policies/comment-policy.yml: trust_critical_globs` — never hardcoded.

## Explicitly NOT done

- **No regex/lint gate.** Distinguishing a tripwire from a restate-WHAT comment is semantic, not pattern-matchable; a regex gate produces false positives that train agents to write worse comments to satisfy it. Enforcement is write-time (the rule) + review-time (code-review).
- **Not retroactive.** Existing files are cleaned only via an explicit Lightweight pass (FR-21). Comments strip from build output, so cleanups need no deploy.
- **No new skill.** The rule's primary home is FLOW_RULES.md (per the proposal's integration points); detail + the audit prompt live in `docs/comment-policy.md`. Skill count unchanged (24).

## Adoption (per project)

1. Run the independent-audit prompt in `docs/comment-policy.md` against your repo.
2. Set the returned carve-out globs in `policies/comment-policy.yml: trust_critical_globs`.
3. The policy is then in force at write-time (FR-22) and review-time (`code-review`).

Rollback: `git revert <SHA>` (one commit).
