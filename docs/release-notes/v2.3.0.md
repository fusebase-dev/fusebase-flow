# Fusebase Flow v2.3.0 — `upgrade-engine.sh` (operator-explicit engine upgrade)

**Released:** 2026-05-10

## What's new

A new operator-maintained script — `hooks/local/upgrade-engine.sh` — that closes the engine-upgrade loop. When upstream ships a new health-check engine version (like v2.2.1's duplicate-marker detection), pulling `.fusebase-flow-source/` updates the upstream-comparison reference but **doesn't** automatically update your local engine. v2.3.0 fixes that.

```bash
# Refresh upstream clone
cd .fusebase-flow-source && git pull origin main && cd ..

# Apply engine + recovery + VERSION sync (interactive, prompts before applying)
bash hooks/local/upgrade-engine.sh
```

## Why this matters

Pre-v2.3.0:

- `mirror-skills.sh` and `mirror-agents.sh` sync canonical `skills/` + `agents/` from upstream → provider mirrors. Auto-discovery in the engine then picks up new skills / agents on next health-check run. Zero maintenance for those.
- But `hooks/local/fusebase-flow-health-check.sh` and `hooks/local/post-fusebase-update.sh` are **operator-maintained scripts**. They deliberately don't auto-mirror because operators may have local customization.
- This created a gap: upgrading the engine + recovery scripts was a manual file-by-file copy. Easy to forget. We hit this exact gap during the v2.2.0 → v2.2.1 transition where downstream test projects had a stale v2.2.0 engine without v2.2.1's duplicate-marker detection.

v2.3.0 closes the gap with an explicit, opt-in script. Operators who DO want the new upstream engine run one command. Operators who've customized their local engine just don't run it.

## What `upgrade-engine.sh` syncs

| File | Reason |
|---|---|
| `hooks/local/upgrade-engine.sh` (itself) | So future runs of this script adopt newer versions of itself seamlessly. Bootstrap is one manual copy; everything after is automatic. |
| `hooks/local/fusebase-flow-health-check.sh` | The engine. Picks up upstream improvements (e.g. v2.2.1 duplicate-marker detection). |
| `hooks/local/post-fusebase-update.sh` | The recovery script. Picks up upstream improvements (e.g. new recovery steps, hardened patches). |
| `VERSION` | Project's record of installed framework version. Bumped to match upstream's. |

## What it does NOT touch

| File | Why |
|---|---|
| `hooks/local/fusebase-flow-overlays/agents-md-overlay.md` | Operator-customizable. Project-specific values table at the bottom. Replacing would clobber customization. |
| `hooks/local/fusebase-flow-overlays/claude-md-overlay.md` | Same. |
| `hooks/local/fusebase-flow-overlays/settings-json-merge.py` | Auto-discovers from upstream at runtime; no benefit to syncing the file. |
| `hooks/local/fusebase-flow-overlays/skills/fusebase-flow-health-check/SKILL.md` | Auto-handled by `mirror-skills.sh`. |
| `hooks/local/fusebase-flow-overlays/commands/fusebase-health.md` | Auto-handled by `post-fusebase-update.sh` Step 10. |
| `skills/`, `agents/` | Canonical content; use `mirror-skills.sh` / `mirror-agents.sh`. |
| `AGENTS.md`, `CLAUDE.md`, `.claude/*` | Managed via `post-fusebase-update.sh`. |

If you want a wider sync (e.g. also overlay templates), you can either edit `FILES_TO_SYNC` in your local `upgrade-engine.sh` or do the cp manually for those specific files.

## Three usage modes

| Mode | Command | Behavior |
|---|---|---|
| Interactive (default) | `bash hooks/local/upgrade-engine.sh` | Prints upstream HEAD + VERSION, lists pending changes with diff stats, prompts `y/N` |
| Non-interactive | `bash hooks/local/upgrade-engine.sh --auto-yes` | Applies without prompt (e.g. for scripted upgrade flows) |
| Preview only | `bash hooks/local/upgrade-engine.sh --dry-run` | Shows what would change; no writes; no prompt |

## Backups

Each replaced file gets a `.pre-upgrade-<UTC-timestamp>` backup. Easy rollback:

```bash
# If something looks wrong after upgrade
mv hooks/local/fusebase-flow-health-check.sh.pre-upgrade-20260510T015800Z hooks/local/fusebase-flow-health-check.sh
```

Once you've validated the upgrade (run the health check, it reports HEALTHY, hook tests pass), you can clean up:

```bash
rm hooks/local/*.pre-upgrade-* VERSION.pre-upgrade-* 2>/dev/null
```

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Success — or "already up to date, nothing to do" |
| 1 | Upstream clone (`.fusebase-flow-source/`) missing or unreadable |
| 2 | Unknown CLI argument |
| 3 | Operator declined at the interactive prompt |

## Notes for upgraders (v2.2.x → v2.3.0)

- **Bootstrap step (one-time per project):** an existing v2.2.x project doesn't yet have `upgrade-engine.sh` locally. Get it by manual copy:
  ```bash
  cd .fusebase-flow-source && git pull origin main && cd ..
  cp .fusebase-flow-source/hooks/local/upgrade-engine.sh hooks/local/upgrade-engine.sh
  chmod +x hooks/local/upgrade-engine.sh
  ```
  After this one-time bootstrap, future engine upgrades (v2.3.1, v2.4.0, ...) are seamless: just `bash hooks/local/upgrade-engine.sh`.
- **Recovery script unchanged.** `hooks/local/post-fusebase-update.sh` ships the same content as v2.2.1.
- **Health-check engine unchanged from v2.2.1.** Still does duplicate-marker detection.

This release is purely additive: nothing existing changes behavior. Pulling v2.3.0 by itself does nothing visible until you run the new script.

## What's next (future releases)

The pattern of v2.3.0 — explicit opt-in operator scripts for things that used to require manual file-by-file copying — could extend to:

- A future `upgrade-overlays.sh` that lets operators sync overlay templates (the ones we explicitly excluded above) with conflict markers for the project-specific values tables. Lower-priority because overlay templates change rarely.
- A future `validate-upgrade.sh` that runs a battery of post-upgrade checks (preflight + hook tests + health check + smoke test that recovery still works).

Both are speculative; we'll add them only if real operators hit those gaps.
