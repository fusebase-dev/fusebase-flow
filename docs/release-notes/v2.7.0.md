# Fusebase Flow v2.7.0 — workflow-mode-aware `artifact_ttl_minutes` for `production_deploy`

**Released:** 2026-05-10

## TL;DR

The `production_deploy.artifact_ttl_minutes` field in `policies/approval-policy.yml` is now **mode-aware**. Solo direct-to-main projects get a long, "cookie-like" TTL by default (90 days); team `branch_pr` projects keep the original 60-min TTL. The reader handles both new mode-keyed object form and legacy flat-int form (backward compatible).

No engine changes. No template changes. No role-discipline changes. One config schema improvement + one 5-line hook update.

## The friction this fixes

Real-world deploy debugging session in `paperclip+hermes-v1`:

| Step | What happened |
|---|---|
| 1 | Operator authored `production_deploy` approval artifact (60 min TTL) |
| 2 | Deploy started, hit cookie capture issue, **aborted** |
| 3 | SSH tunnel wedged, **second abort** |
| 4 | VS Code zombie IPv4 listener captured a fake cookie, **third abort** |
| 5 | Approval artifact expired (60 min was up); operator had to re-author and burn time on policy busywork instead of debugging the actual issue |

The downstream PO diagnosed it correctly: **DP.6 is the real gate**. The `APPROVE-DEPLOY-NOW` magic phrase typed at deploy time in the Deploy session is non-delegable and non-bypassable. The artifact's TTL only serves stale-state protection. For one operator iterating on one machine, that protection is doing very little incremental work above DP.6.

But for team contexts (multiple operators, days-old PR reviews, possibly compromised CI runners), the 60-min TTL provides real stale-state protection. Both defaults are correct — for their respective modes.

## The fix — mode-aware TTL

```yaml
require_approval:
  production_deploy:
    enforce: true
    artifact_ttl_minutes:
      direct_to_main: 129600   # 90 days; cookie-like; DP.6 is the real gate
      branch_pr: 60            # 60 min; stale-state protection for team contexts
    rationale: "Production deploys are externally visible and irreversible without rollback ceremony. DP.6 magic phrase is the real per-deploy gate; TTL is stale-state protection only — long for solo direct-to-main, short for team branch_pr."
```

When `hooks/local/approve-local.sh` writes a new artifact, it:

1. Reads `policies/approval-policy.yml`
2. Looks up `require_approval.production_deploy.artifact_ttl_minutes`
3. If it's an integer → use as-is (legacy v1 form)
4. If it's a dict → look up by `workflow_mode` field; fall back to `direct_to_main` if mode missing; fall back to 60 if both missing
5. Computes `expires_at = now + TTL_minutes` and writes into the artifact

## Backward compatibility — strict superset

| Project shape | What happens on v2.7.0 |
|---|---|
| Flat int `artifact_ttl_minutes: 60` (legacy v1 schema) | Used as-is. No change in behavior. |
| Flat int set to a custom value (e.g., operator overrode to `129600`) | Used as-is. Local override respected. |
| Mode-keyed object (new v2 form) | Reader looks up `workflow_mode` and applies matching value. |
| Local override in `policies/approval-policy.local.yml` (gitignored) | Layered on top; takes precedence; either shape works. |

`schema_version` field bumped from `1` to `2`. Both schemas continue to work.

## Default values

| Mode | Default TTL | Rationale |
|---|---|---|
| `direct_to_main` | **129600 min (90 days)** | Cookie-like; matches the manual override pattern operators have been using. DP.6 is the real per-deploy gate. Solo iteration / multi-attempt debugging doesn't burn through the window. |
| `branch_pr` | **60 min** (unchanged) | Team context; stale-state protection real; original rationale stands. |

## Migration paths

### Project on v2.6.1 with upstream default

Pull `policies/approval-policy.yml` from v2.7.0 upstream:

```bash
cd .fusebase-flow-source && git pull origin main && cd ..
cp .fusebase-flow-source/policies/approval-policy.yml policies/approval-policy.yml
```

Your `direct_to_main` mode automatically gets the 90-day default. No further action needed.

### Project on v2.6.x with manual flat-int override (e.g., 129600)

Two options:

1. **Keep your local override.** Continues to work. Reader uses the flat int regardless of mode.
2. **Migrate to mode-keyed form.** Replace your flat-int with the mode-keyed object form. Cleaner semantics; if you ever switch to `branch_pr` mode, you get the right TTL automatically.

### Project on `branch_pr` mode

Pull upstream. Your TTL stays at 60 min (mode-aware default for `branch_pr`). No behavior change.

### Project with `policies/approval-policy.local.yml` override

Local override is gitignored and takes precedence over the committed default. Whatever shape you used in the local file (flat int or mode-keyed) continues to work.

## Backward-compat invariants

- **No project on v2.6.x or older breaks.** Flat-int form still reads correctly.
- **No security regression.** DP.6 magic phrase remains the real per-deploy gate. Long TTL only relaxes stale-state protection, not authorization.
- **`policies/approval-policy.local.yml` mechanism unchanged.** Operators can still override per-project with either shape.
- **All other `require_approval.<action>.artifact_ttl_minutes` fields unchanged** — `database_migration` (120 min), `destructive_file_delete` (30 min), etc. all stay flat-int. Only `production_deploy` gets the mode-aware shape because it's the only operation with the DP.6-equivalent gate dynamic.

## Why ship as v2.7.0 (minor) not v2.6.2 (patch)

This adds a new schema feature (mode-keyed TTL form) and a new default value pattern. That's a new framework capability, not a bug fix — minor version reflects the additive scope.
