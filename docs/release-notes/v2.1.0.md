# Fusebase Flow Local — v2.1.0 release notes

**Release date:** 2026-05-09
**Previous version:** 0.1.2 (cumulative changelog: `CHANGELOG.md`)
**Tag:** `v2.1.0`

## What's new

v2.1.0 adds **two role-shaped sub-agents** — Product Owner and AI Developer — that together cover the full Fusebase Flow eight-phase ticket lifecycle. They are an opt-in entry point on top of the existing skills + workflows + hooks; the framework remains fully usable without them.

The two-agent design intentionally absorbs Architect responsibilities into the Product Owner so operators only need to context-switch between **two** agent identities (PO and AI Developer) instead of four (PO, Architect, Implementer, Deploy). The AI Developer is parameterized by the handoff filename it is given — `*-implement.md` makes it self-attest as Implementer; `*-deploy.md` makes it self-attest as Deploy phase.

## How operators use sub-agents

### Claude Code

```
> Use the product-owner sub-agent. Let's ship pagination.
```

Claude Code auto-discovers `.claude/agents/product-owner.md`. The sub-agent self-attests, loads the required reads, and walks you through Specify → Clarify → Plan → Decisions → Tasks → draft-verification-gate. It saves a handoff at `docs/handoff/<date>-pagination-implement.md` before chat output (FR-04).

Then in a fresh session:

```
> Use the ai-developer sub-agent. Run docs/handoff/2026-05-09-pagination-implement.md.
```

The AI Developer reads the handoff, self-attests as Implementer, executes the T-chain, stops at the verification gate, and produces the gate report.

### Codex

Codex does not auto-discover sub-agent files. Operators reference them in the first message of a fresh session:

```
> Read .codex/agents/product-owner.md and operate as Product Owner per its instructions.
> Let's ship pagination.
```

Then in another fresh session:

```
> Read .codex/agents/ai-developer.md and operate as AI Developer per its instructions.
> Run docs/handoff/2026-05-09-pagination-implement.md.
```

Same eight-phase flow, same role-discipline guardrails, same handoff-on-disk discipline. Provider parity is preserved.

## Tool surfaces

### Product Owner

| Allowed | Use |
|---|---|
| Read, Glob, Grep | every file in repo |
| Bash | read-only investigation (`git status`, `git diff`, `git log`, `ls`, `cat`, etc.) |
| Write, Edit | scoped to `docs/specs/`, `docs/decisions/`, `docs/backlog/`, `docs/handoff/`, `docs/problem-catalog/` only |
| AskUserQuestion | clarify Q&A; recommendations with 2–3 options + tradeoffs |

| Denied | Why |
|---|---|
| Edit/Write to application code, `hooks/`, `policies/`, `workflows/`, `templates/`, `skills/`, `audit/` | PO.1, PO.2 — PO doesn't write code or framework files |
| `git push`, `git commit` of code | AI Developer commits T-task work; PO commits only the final docs bundle at 8c |
| Approve deploy without verification-gate evidence | PO.3 |
| Lock a decision the operator hasn't confirmed | PO.5 |

### AI Developer

| Allowed | Use |
|---|---|
| Read, Glob, Grep | investigate repo |
| Bash | build / test / lint / `git status` / `git diff` / `git log` / `git add <specific-file>` / `git commit` / `git push` to feature branch |
| Write, Edit | application code, test files, framework hooks/policies if the ticket explicitly scopes it |
| AskUserQuestion | rare — when handoff is ambiguous |

| Denied | Why |
|---|---|
| Drafting `spec.md` / `decisions.md` / `tasks.md` / `verification-gate.md` | PO's job |
| Modifying LOCKED `decisions.md` | IM.2 — STOP and surface to PO |
| `rm -rf`, `git push --force`, `git reset --hard`, `git add -A`, `--no-verify` | FR-06 / IM.6 — already deny-listed in `policies/command-policy.yml` |
| Running deploy without a deploy handoff | IM.1 / DP.1 |
| Marking spec DONE without deploy hash or on probe failure | DP.3 / DP.5 |

## Hard rails (defense in depth)

| Risk | First-line guard (prompt) | Second-line guard (framework) |
|---|---|---|
| PO writes app code | PO.1 in role-discipline; "Denied" table in PO `AGENT.md` | Relies on prompt + role-discipline (no hook-level role × path enforcement yet) |
| AI Dev drafts a spec | "Denied" table in AI Developer `AGENT.md` | role-discipline IM section |
| AI Dev runs deploy without handoff | "STOP and ask" rule when no handoff path provided | `policies/required-artifacts.yml: before_deploy_complete_claim` + Stop hook |
| AI Dev runs `git push --force` / `--no-verify` | IM.6 in role-discipline | `policies/command-policy.yml` deny-list; pre-commit + `pre_tool_use` hooks block |
| Cookie / session-key persisted | IM.7; `workflows/live-user-verification.md` | `policies/secret-patterns.yml: cookie_session_value` blocks `pre_tool_use` + `git_pre_commit` |
| Locked decision modified mid-implement | IM.2 STOP-and-surface | None at hook level (intentional — surfaces to PO) |
| Sub-agent doesn't load `FLOW_RULES.md` at start | "Required reads at session start" table in each `AGENT.md` | `hooks/handlers/session_start.py` `REQUIRED_TOP_FILES` enforcement |
| Agent files drift between providers | `bash hooks/local/mirror-agents.sh` | preflight step 5b warns on drift |

## Upgrade notes

- **Self-attestation phrasing changed.** Sessions that started before the upgrade may still emit `Fusebase Flow v0.1`; restart the session to pick up the v2.1 phrasing. The framework rules themselves (FR-01..FR-15) are unchanged from v0.1 — the bump reflects the sub-agents addition and the major-version stability commitment.
- **Sub-agents are opt-in.** You can keep using the framework via the existing skills + workflows pattern without ever invoking a sub-agent. Sub-agents are an additional entry point.
- **Mirror script now has a sibling.** `hooks/local/mirror-agents.sh` is parallel to `mirror-skills.sh`. After updating either canonical (`skills/<name>/SKILL.md` or `agents/<name>/AGENT.md`), run the corresponding mirror script. Preflight will warn on drift.
- **Install.sh prompts changed.** `1/3..3/3` → `1/4..4/4`. Step 4 mirrors agents.
