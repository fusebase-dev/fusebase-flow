# Fusebase Flow v3.8.5 — `--wire-hooks` wires stop.py correctly onto an existing CLI Stop chain (U14)

**Release date:** 2026-06-01
**Previous version:** 3.8.4

## Summary

A downstream reported (and we reproduced) that `post-fusebase-update.sh --wire-hooks`, on a project whose `.claude/settings.json` already contained CLI Stop hooks, produced a Stop entry **labeled** as the Flow hook but carrying the **CLI** `run-typecheck-apps.js` command. Result: Flow's `stop.py` was never wired (end-of-turn enforcement silently off) and a CLI typecheck ran twice. 5 of 6 lifecycle events wired correctly — only the shared Stop event was wrong.

## Fixed

### U14 — discovery picked `handlers[0]` instead of the Flow handler

`settings-json-merge.py` auto-discovers each event's command from the upstream `.fusebase-flow-source/.claude/settings.json.example`. It read `handlers[0].command`:

- For **Flow-only** events (SessionStart, PreToolUse, …) `handlers[0]` is the Flow handler → correct.
- For the **shared Stop** event, the example chain is `[run-typecheck-apps.js, quality-check-apps.js, stop.py]`, so `handlers[0]` is a **CLI** command. That got discovered as the "Stop" Flow command and appended under the Flow label — and `stop.py` was never wired.

The fix: discovery now scans each event's chain for the **Flow** handler (command under `hooks/handlers/`) and uses that, falling back to `handlers[0]` only when none match. The Stop event now resolves to `stop.py` regardless of how the CLI hooks are ordered in the chain.

Why the existing test didn't catch it: the recovery sim's `--wire-hooks` case runs in a temp project **without** a `.fusebase-flow-source/`, so discovery fell back to the hardcoded `stop.py` default and looked fine. The bug only appears when discovering from the upstream example (the real upgrade path).

### Regression test (U14)

`test-cli-flow-recovery.sh` now merges onto a `.claude/settings.json` that already has CLI Stop hooks **with an upstream example present**, and asserts:
- the Flow-labeled Stop entry's command is `…/hooks/handlers/stop.py` (not a CLI command),
- `stop.py` is present in the Stop chain,
- the CLI typecheck hook is preserved exactly once (not duplicated, not dropped).

## Note for downstreams already on ≤3.8.4

If you ran `--wire-hooks` on a project with pre-existing CLI Stop hooks and hand-edited `settings.json` to wire `stop.py`, that's correct and stays. Re-running `--wire-hooks` on 3.8.5 is idempotent and will wire `stop.py` properly on a fresh settings.json. (Hook wiring remains opt-in per F3/U11.)
