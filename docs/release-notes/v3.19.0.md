# Fusebase Flow v3.19.0 — app-quality-patterns (cross-project behavioral quality library)

**Release date:** 2026-06-11
**Previous version:** 3.18.2

## Summary

Operator-driven: the same behavioral defects recur across consumer projects — filters/reports not encoded in the URL (refresh or share loses the view), deletes leaving orphaned records, misaligned dropdown chevrons. LLMs know these requirements; they just don't apply them unless the requirement is in context at the right lifecycle moment. v3.19.0 ships the carrier: a curated, growing pattern library whose entries become **spec acceptance criteria by ID** — riding the existing tasks → gate → smoke → review machinery with zero new gates.

## The library

`flow-skills/app-quality-patterns/` — thin router + three lazy-loaded category files. **14 seeded patterns**, each `ID · Trigger · Requirement · Verify (copy-ready smoke recipe) · Anti-pattern`:

| Category | Patterns |
|---|---|
| State & navigation (QP-01..04) | URL reflects view state (refresh/share restores it) · back/forward work · deep links guarded (404/403 designed) · list position survives back-navigation |
| Data integrity (QP-10..14) | every delete names its cascade policy, zero orphans (+ second-path probe) · mutations invalidate dependent views · double-submit guarded · optimistic updates roll back visibly · destructive actions confirm scope or offer undo |
| UI polish (QP-20..24) | chevron alignment/rotation truth · empty/loading/error states designed · form validation preserves work · timezone-correct rendering · unsaved-changes guard |

## How it reaches the LLM (the delivery design)

| Phase | Mechanism |
|---|---|
| Specify | `requirements-specification` scans the category index on app-feature tickets → matching patterns become **ACs citing QP IDs** |
| Plan | design brief lists the applicable IDs; implementer reads the cited category |
| Gate/Smoke | each pattern's Verify line is a copy-ready S\<n\> recipe (e.g., "delete parent → query store → assert zero orphans") |
| Review | `code-review` checks QP-ACs are honored — semantic, by reading |

Why this shape (lessons from the v3.15–v3.18 line): always-on lists tax every session; description-matched skills alone miss sessions; behavioral requirements aren't regex-able — but a spec AC survives the whole pipeline on existing machinery. Boundary: QP owns **WHAT must be true**; CLI skills (`app-ui-design`, …) own **HOW on the stack** (overlap-map row added).

## Growth

One defect observed across ≥2 projects = one new table row, next release. Project-specific patterns live in that project's `docs/skills/`, never in canonical.
