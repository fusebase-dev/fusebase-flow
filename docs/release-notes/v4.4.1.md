# Fusebase Flow v4.4.1 — upgrade backups no longer block `fusebase update`

**Released:** 2026-07-14
**Type:** patch (field escalation; unblocks the platform update flow)

## Why

A downstream consumer (Windows 11 / Git-Bash, FuseBase CLI edition) upgraded 3.30.2 → 4.3.2 and then found **`fusebase update` blocked by Flow itself**. The CLI's "create a pre-update Git checkpoint commit? → Yes" runs `git add -A` + commit; Flow's pre-commit secret scan then hard-BLOCKED it with:

```
[fusebase-flow:pre-commit] BLOCK — secret pattern in staged added lines:
  github_personal_access_token (high) — redacted
  anthropic_api_key (high) — redacted
  cookie_session_value (medium) — redacted
```

There was **no real credential**. The hits came from Flow's own upgrade **backups** — specifically the `hooks.pre-upgrade-<ts>/` snapshot (carrying the OLD secret-scan **test fixtures** — deliberate dummy `ghp_…`/`sk-ant-…`/cookie literals) and `policies.pre-upgrade-<ts>/secret-patterns.yml`. (Flow also drops `*.pre-bootstrap-<ts>` and `AGENTS.md`/`CLAUDE.md` `.pre-refresh-<ts>` backups, but those don't carry the designed fixtures.) The scanner excluded the LIVE fixture/policy paths but not their renamed backup twins. Worse, after the blocked commit the fixture blobs stayed **staged** (`AD`), so every later commit kept blocking until someone knew to `git restore --staged` them.

## The fix (two layers + UX)

1. **Scanner excludes ONLY the fixture/policy backup twins.** `hooks/shared/staged_secret_scan.py` excludes root-anchored `hooks.pre-upgrade-<ts>/tests/fixtures/**` and `policies.pre-upgrade-<ts>/secret-patterns*.yml`, where `<ts>` is the EXACT `[0-9]{8}T[0-9]{6}Z` stamp — **not** a broad `*.pre-*`, and **not** a loose `*T*Z` (which matches literal `TZ` at any depth); both are scanner bypasses (caught in Codex-xHigh review). These twins are copies of the SAME files the live D-A1 excludes already exempt (`hooks/tests/fixtures/`, `secret-patterns.yml` — the documented D-A1 gap, where a real secret is likewise not caught), so this extends that existing exemption to their upgrade-backup copies — **not** a new content category. Pathspec matches by name only, so the exact `[0-9]{8}T[0-9]{6}Z` stamp + root-anchoring keep the exempted set exactly as narrow as the live D-A1 set. A backup of any other file, a `TZ`/non-root/wrong-prefix look-alike, and any untimestamped name all stay **scanned**. The installed `hooks/git/pre-commit` block message was updated to match.
2. **Backups are git-excluded at creation.** `upgrade.sh`, `bootstrap-upgrade.sh`, and `post-fusebase-update.sh` append EXACT-timestamp patterns to `.git/info/exclude` (via `git rev-parse --git-path`, worktree-correct; ensuring a trailing newline first so an unterminated line can't swallow the rule) **before** writing backups, so `git add -A` / the CLI checkpoint never stages them (which also prevents the `AD`-staged trap). Per-pattern idempotent, `set -e`-safe, and honest on failure (WARN + a conditioned "git-excluded" note). Keep backups until validated — only previously-committed content is recoverable from git history.
3. **Accurate guidance.** The BLOCK message no longer waves through a `.pre-*` hit as "not a real secret" (only the exact fixture/policy twins are skipped) — it says **rotate if it is real**, and separately `git restore --staged` any Flow backup blobs a pre-fix `git add -A` left staged (they should never be committed). `upgrade.sh`'s final notes say the same, condition their "git-excluded" claim on whether `.git/info/exclude` was actually written, and add: **answer "No" to the CLI checkpoint prompt mid-ticket** so WIP growth in grandfathered over-ceiling files doesn't separately trip the FR-25 ratchet on a wholesale add.

## If you're already stuck (existing v4.3.2/v4.4.0 backups)

The fix ships in the scanner + upgrade tooling, but a repo that already has ungitignored backups from an earlier upgrade can unblock. **Review before deleting** — don't blanket-`rm` a `.pre-*` glob (that would also delete a legit `config.pre-upgrade-template.yml` and miss nested file backups). First list the exact timestamped backups and what's staged:

```bash
# genuine Flow backups (exact YYYYMMDDTHHMMSSZ stamp), tracked-or-untracked:
git ls-files -oc | grep -E '\.pre-(upgrade|bootstrap|refresh)-[0-9]{8}T[0-9]{6}Z(/|$)'
# any Flow backup blobs left staged by a pre-fix `git add -A`:
git diff --cached --name-only | grep -E '\.pre-(upgrade|bootstrap|refresh)-[0-9]{8}T[0-9]{6}Z(/|$)'
```

Unstage those blobs (they should never be committed), then remove the paths you've confirmed are genuine backups:

```bash
git restore --staged <paths from the second command>
rm -rf <paths you confirmed are Flow backups>
```

If the secret scan still blocks on a path that is **not** one of those exact `hooks.pre-upgrade-<ts>/tests/fixtures/**` or `policies.pre-upgrade-<ts>/secret-patterns*.yml` twins, treat it as **potentially real** — inspect it, and rotate only if it is real (it could instead be a Flow backup you should unstage, not commit). The scanner does NOT wave through arbitrary `.pre-*` names. See `docs/problem-catalog/upgrade-backups-trip-secret-scan/`.
