# Fusebase Flow v2.6.1 — `.gitignore` exception for `health_check_deferral-*.json` (closes B5)

**Released:** 2026-05-10

## TL;DR

One-line `.gitignore` change. Adds a narrow exception so `health_check_deferral-*.json` artifacts (90-day documents-of-record from v2.4.0) are auto-tracked in git. Other `state/approvals/*` artifacts (notably `production_deploy-*.json` — 60-min ephemeral auth tokens) stay gitignored.

No engine changes. No template changes. No role-discipline changes. Pure `.gitignore` + one doc note.

## Who needs to upgrade

| Environment | Action |
|---|---|
| Projects on v2.6.0 that already added the exception manually | Optional — pulling v2.6.1 makes upstream byte-equivalent to your local; you can keep your manual addition or remove it after pull |
| Projects on v2.6.0 without the exception | Upgrade recommended next time you `bash hooks/local/upgrade-engine.sh` — but `.gitignore` isn't synced by `upgrade-engine.sh` (it only touches `hooks/local/`), so for `.gitignore` you either copy from upstream manually or wait until your next install/setup runs |
| Projects on pre-v2.6.0 | Upgrade through normal multi-version path; v2.6.1 is no harder to land than v2.6.0 |

## Before / after

### Pre-v2.6.1

```
$ cat .gitignore
state/approvals/*
!state/approvals/.gitkeep
```

`git add state/approvals/health_check_deferral-install-discipline-20260510.json` silently no-ops. Operator either force-adds (`git add -f`) per artifact, or modifies `.gitignore` per project.

### v2.6.1

```
$ cat .gitignore
state/approvals/*
!state/approvals/.gitkeep
# health_check_deferral-*.json artifacts are documents of record (90-day TTL,
# operator-authored deferral decisions). They MUST be in git so fresh clones
# reproduce the EXCEPTION_IN_EFFECT verdict and PR review can audit which
# deferrals are active. Other state/approvals/* artifacts (e.g.
# production_deploy-*.json — 60-min ephemeral auth tokens) stay gitignored.
# See docs/health-check-deferrals.md and docs/release-notes/v2.6.1.md.
!state/approvals/health_check_deferral-*.json
```

`git add state/approvals/health_check_deferral-*.json` works as expected. `production_deploy-*.json` and any future ephemeral artifact families remain ignored.

## Why narrow

The exception is intentionally specific to the `health_check_deferral-*.json` glob, not the wider `*.json` glob. Three categories of artifact live in `state/approvals/`:

| Category | TTL | Should be in git? | Why |
|---|---|---|---|
| `health_check_deferral-*.json` | 90 days | **yes** | Document of record; auditable; needs to reproduce on fresh clones |
| `protected_path_edit-*.json` | varies | maybe — case by case | Authorize hook-test paths; sometimes worth tracking, sometimes ephemeral |
| `production_deploy-*.json` | 60 min | **never** | Ephemeral auth-token gate; security-sensitive |

A wide `!state/approvals/*.json` exception would auto-track `production_deploy-*.json` (security risk) and `protected_path_edit-*.json` (sometimes wrong). The narrow form forces every new artifact family to be a deliberate decision: "do we want this in git? Yes → add another negation line. No → it stays ignored."

## Upgrade

```bash
cd .fusebase-flow-source && git pull origin main && cd ..
bash hooks/local/upgrade-engine.sh   # syncs hooks/local/ — note: NOT .gitignore
```

`upgrade-engine.sh` only touches `hooks/local/`. The `.gitignore` change in v2.6.1 lives in the project root and isn't synced. **Three options for picking it up:**

1. **Manual copy:** `cp .fusebase-flow-source/.gitignore ./.gitignore` (carefully — your project may have its own additions; review the diff first).
2. **Selective merge:** open `.fusebase-flow-source/.gitignore`, find the new exception block (around line 7-13), copy that block into your project's `.gitignore` at the equivalent spot.
3. **Skip:** if you already added the exception manually per the per-project workaround that's been documented since v2.4.0, you don't need to do anything. Your local exception matches upstream's now.

For new projects (`fusebase init` or `bash install.sh`), v2.6.1's `.gitignore` is what gets installed — so any project initialized post-v2.6.1 has the exception baked in by default.
