# Fusebase Flow v2.8.0 — FR-17 forward momentum + IM.11 net-time tracking

**Released:** 2026-05-10

## TL;DR

Two operator-experience improvements bundled into one minor release:

1. **FR-17 (anti-retreat).** New always-on rule: agents never suggest closing the session, "letting it bake," resting, postponing, or wrapping up. Every turn ends with the next forward action. If nothing's pending, the agent states "no pending action — your call on what's next" neutrally and lets the operator decide.

2. **IM.11 (per-task wall-clock recording).** AI Developer records UTC `started_at` and `committed_at` for each task. Wall-clock per task = `committed_at − started_at` = net active development time (wait-for-operator time is naturally excluded since it happens between tasks). Both timestamps flow into gate reports + deploy reports for retrospective analysis.

No engine changes (7th release in a row identical to v2.4.1). No policy schema changes. No new templates. Pure rule + measurement + template-field additions.

## What you'll notice as an operator

### FR-17 in action

Before v2.8.0, agent turns frequently ended with phrases like "I'd stop here," "let it bake before iterating," "ready to close?", "save it for tomorrow." That ambiguity is exactly the problem: **operator reads "let's close" and thinks the work is done; the agent meant "I'm being cautious."**

Post-v2.8.0, those phrases are forbidden. Every turn presents the actual next forward action.

| Before | After |
|---|---|
| "I'd recommend we close here and observe before iterating." | "Next forward action: paste the gate report block into PO chat." |
| "Ready to wrap up?" | "Reply with `A`, `B`, or `C` to lock decision X." |
| "We've shipped enough for one day." | "Switch to the Deploy session and type `APPROVE-DEPLOY-NOW`." |
| "Save it for tomorrow when you're fresh." | "No pending action — your call on what's next." (if genuinely nothing) |

The distinction is operator-decision-pre-emption (forbidden) vs operator-decision-handoff (required). If the operator didn't ask "should I stop?", the agent doesn't suggest stopping.

### IM.11 in action

Every gate report now carries per-task timing data:

```
| Task | Title  | Commit SHA | Started (UTC) | Committed (UTC) | Wall-clock | ... |
|------|--------|------------|---------------|-----------------|------------|-----|
| T57  | ...    | abc1234    | 14:02:11      | 14:18:43        | 16:32      | ... |
| T58  | ...    | def5678    | 15:01:22      | 15:09:48        | 8:26       | ... |
| T59  | ...    | 901234a    | 15:42:08      | 15:55:11        | 13:03      | ... |
```

Plus a Time totals section:

```
| Metric                     | Value     | Notes |
|----------------------------|-----------|-------|
| Total elapsed (wall)       | 1:53:00   | end-to-end including all waits |
| Total active development   | 0:38:01   | sum of wall-clocks |
| Wait time (elapsed-active) | 1:14:59   | operator feedback pauses |
```

Wait time happens between tasks (operator review, decisions). Active time happens within tasks (agent actually working). Operator can spot patterns: "task X consistently takes 20 min" or "wait time is dominating the cycle" or "deploy command took 4 min today vs 30s last week — something's off in the build pipeline."

## Anti-pattern catalog (forbidden in v2.8.0+)

The role-discipline skill ships a concrete catalog. Phrases the agent must delete from drafts before sending:

- "Let it bake."
- "Save it for tomorrow."
- "Close session?"
- "Ready to wrap up?"
- "I'd stop here."
- "We've shipped enough for one day."
- "Premature optimization — observe first."  *(when unprompted)*
- "Don't push for it."
- "Time to rest."
- "Take a break before iterating."
- "Your day's been productive — close it out."
- "Pause and observe."

Edge case: in response to a direct question ("should we ship v2.X.Y now?"), agent can recommend waiting if the engineering judgment is sound. That's advice + a forward action. **Unprompted retreat suggestions** are what FR-17 forbids.

## Drivers (operator-surfaced friction, 2026-05-10)

### FR-17 driver

> "AI always tries to avoid continue working[, tries to make the] operator stop. It constantly engages in things like:
> - You are done.
> - Go to rest.
> - Let's postpone.
> - Let's close the day.
>
> This is not productive, and we need to exclude this behavior. The goal of the product owner and AI developer is to move forward to close the project. So we need to exclude such behavior because it's unnecessary and confuses the operator. Because the operator thinks that all was done, but in the same case AI just tries to postpone things."

The framework's previous absence of an anti-retreat rule let agents drift into wrapping-up phrasing by default. FR-17 codifies the reverse: forward momentum is the rule, not the exception.

### IM.11 driver

> "AI developer reports some of the things. Let's add also the time which was taken to execute the task. Excluding the wait time when the AI developer waits for the feedback or something, we need to check the net time of actual development. That's going to help in the future to do retrospective analysis and improve the flow."

Pre-v2.8.0, gate and deploy reports captured timestamps but didn't surface aggregate active-development time. Without that data, retrospective analysis ("are we getting faster? where's wait time concentrated?") was qualitative. IM.11 makes it quantitative.
