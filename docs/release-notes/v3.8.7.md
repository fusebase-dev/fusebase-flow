# Fusebase Flow v3.8.7 — install/upgrade review against v3.8.5 (Windows overlay)

**Release date:** 2026-06-01
**Previous version:** 3.8.6

## Summary

A downstream verified v3.8.4 (Issue 2) and v3.8.5/U14 as fixed, and surfaced five more findings (two High) from an in-place v3.8.3 → v3.8.5 upgrade into an existing FuseBase CLI project on Windows. All confirmed against the cited locations and fixed.

## Fixed

### F1 [High] — install doc omitted `agents/` from the additive-copy list

`docs/install-fusebase-cli-project.md` "Safe additive copies" listed `skills/ workflows/ policies/ templates/ hooks/ audit/ state/` but **not `agents/`** — yet `hooks/local/mirror-agents.sh` requires the canonical `agents/` tree (the `ai-developer` / `product-owner` sub-agents). Following the doc literally → mirror-agents aborts, `.claude/agents/` never populated, health → `FLOW_LAYER_DRIFT` (0/2 sub-agents). **Fix:** added `agents/` to the list and both the bash and PowerShell copy blocks, with a note on why it's required.

### F2 [High] — U11 (hooks-off ≠ drift) was only half-applied

U11 (v3.8.3) made the **conflict checker** treat a deliberate hooks-off install as benign — but the **main** health engine (`fusebase-flow-health-check.sh`) independently `record_drift`-ed the same state. So an overlay-only opt-in install (the documented default — CLI hooks present, no Flow events) returned `SHARED_MERGE_DRIFT` even while the ownership report said HEALTHY, and the printed remediation (`post-fusebase-update.sh`) wires hooks — defeating "opt-in."

**Fix:** the main engine is now U11-consistent. With CLI hooks present:
- all Flow events wired **+ `stop.py` present** → OK;
- all events present **but `stop.py` missing** → drift (genuine **U14**-style mis-wire);
- `stop.py` present **but events incomplete** → drift (genuine partial degradation);
- **no `stop.py` and events not wired** → **benign opt-in** (LOCAL_OK), *not* drift — "Flow lifecycle hooks not wired (opt-in default; enable with `--wire-hooks`). Existing CLI hooks preserved."

Behavioral regression test **U16** runs the actual main engine on a hooks-off overlay and asserts the verdict is **not** `SHARED_MERGE_DRIFT`.

### F3 [Med] — `.gitattributes` & friends removed from the unconditional copy list

Flow's `.gitattributes` carries repo-wide `* text=auto` + `eol=lf`. Copied into an existing (especially Windows) repo, the next checkout/add **renormalizes line endings across every file** → massive spurious diff + polluted blame. **Fix:** `.gitattributes`, `LICENSE`, `PUBLISHING.md`, and `.python-version` are removed from the blind-copy list and moved to a new **"Copy only after review"** section explaining each risk (`.gitattributes` eol bomb; `LICENSE` overwrites yours; `PUBLISHING.md` is Flow-internal; `.python-version` pins Python).

### F4 [Low] — upstream comparison misreported on a shallow/tag staging clone

`bootstrap-upgrade.sh` clones `.fusebase-flow-source` with `--depth 1 --branch <ref>`. Such a clone can't resolve `origin/main` or traverse history, so the health engine printed a spurious "upstream NEWER … behind by ? commits." **Fix:** the engine now detects a shallow/detached/unresolvable state and prints "upstream comparison unavailable (shallow/tag staging clone — no resolvable origin/main). Staged source VERSION: `<x>`. For a precise comparison: `git fetch --unshallow origin main`, or re-clone without `--depth`." The genuine ahead/behind path is unchanged for full clones.

### F5 [Low] — documented (no code change)

`--wire-hooks` injects the canonical **node** Stop hooks (`run-typecheck-apps.js`, `quality-check-apps.js`). If a project also wires a deprecated `*-on-stop.sh` duplicate, both run on Stop (double typecheck). This is intended (node hooks are canonical; the jq/bash duplicates are deprecated and not auto-re-injected) — now documented so it isn't surprising.

### Out of scope (FuseBase CLI repo)

The CLI's `project-template/eslint.config.mjs` ignores `.claude/**` but not `.codex/**`, while the CLI emits `require()`-style `.codex/hooks/*.js` → `npm run lint` fails. That's a CLI-template issue, not Flow (Flow doesn't own `.codex/hooks` or the eslint config). Routed to the CLI repo. (Flow's own staging-clone lint blocker was fixed separately in v3.8.6.)
