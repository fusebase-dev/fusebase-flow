# Fusebase Flow Local — v2.1.1 release notes

**Release date:** 2026-05-09
**Previous version:** 2.1.0 (cumulative changelog: `CHANGELOG.md`)
**Tag:** `v2.1.1`

## What's new

v2.1.1 ships two **defense-in-depth refinements** to the v2.1.0 sub-agent design, both moving guarantees from prompt-level (LLM judgment) to structural (tool/control-flow). Origin: independent v2.1.0 evaluation feedback that surfaced two boundary-fuzziness failure modes.

| Refinement | What it closes |
|---|---|
| `hooks/local/po-investigate.sh` wrapper | The "read-only Bash" boundary on the PO sub-agent was prompt-level; an LLM had to police `git stash` (mutates), `npm view` (network), `node -e "fs.readFileSync(...)"` (one keystroke from `writeFileSync`) on every shell call. Now structural. |
| `DP.6` deploy-time operator confirm | Production cutovers via the AI Developer Deploy phase had no human-at-the-keyboard moment. Probe-fail latency could be minutes if the operator's attention was elsewhere. Now ~5s structural friction. |

Neither refinement changes v2.1.0's architecture. Two sub-agents, role-discipline-driven, handoff-on-disk — same shape, more layered defense.

## PO investigation wrapper (Deviation 1)

### Before v2.1.1

```yaml
# agents/product-owner/AGENT.md (v2.1.0)
tools: Read, Glob, Grep, Bash, Write, Edit, AskUserQuestion
```

System-prompt instruction: "Bash for read-only investigation only — `git status`, `git diff`, `git log`, `ls`, `cat`, `head`, `tail`, `find`."

The boundary depended on the LLM correctly classifying every shell call. Slippery edges:

| Command | Read-only? | Decision required |
|---|---|---|
| `git status` | yes | allow |
| `git stash` | no — mutates | should refuse — will the LLM remember? |
| `npm view <pkg>` | network call | edge case |
| `npm install` | mutates | should refuse |
| `node -e "fs.readFileSync('foo')"` | yes | allow? but the same syntax mutates with `writeFileSync` |
| `bash -c 'cat foo'` | yes | allow? but the same wrapper enables anything |

### After v2.1.1

```bash
# PO must run shell-backed investigation through the wrapper:
bash hooks/local/po-investigate.sh status
bash hooks/local/po-investigate.sh diff src/dashboard/pagination.tsx
bash hooks/local/po-investigate.sh log --oneline -10
bash hooks/local/po-investigate.sh show abc1234
bash hooks/local/po-investigate.sh blame src/auth/middleware.ts
bash hooks/local/po-investigate.sh cat docs/specs/pagination/spec.md
```

| Allowlisted subcommand | What it does |
|---|---|
| `status` | `git status --short` |
| `diff [args]` | `git diff [args]` |
| `log [args]` | `git log [args]` |
| `show [args]` | `git show [args]` |
| `blame <path>` | `git blame <path>` |
| `ls [path]` | `ls -la [path]` |
| `cat <path>` | `cat <path>` |
| `head <path>` | `head -200 <path>` |
| `tail <path>` | `tail -200 <path>` |
| `find <name-pattern>` | `find . -name <pattern> -type f` (no `-exec`, no `-delete`) |

Anything else exits non-zero with `unknown or non-allowlisted subcommand`. Mutating commands aren't reachable because they aren't subcommand keys.

### Updated PO tool-surface row

```diff
- | Bash | read-only inspection (git status, git diff, git log, ls, cat, head, tail, find) |
+ | Bash | invoke ONLY `bash hooks/local/po-investigate.sh <subcommand> [args]` for shell-backed investigation. Do NOT call `git`, `npm`, `node`, `python`, `cat`, `head`, `tail`, `find`, etc. directly via Bash. The wrapper exposes an allowlist of read-only subcommands; anything else is structurally rejected (exit 2). |
```

The PO Denied table also picks up an explicit row prohibiting direct Bash calls outside the wrapper.

## Deploy-time operator confirm (Deviation 2)

### Failure mode v2.1.0 hits

```
T+0:    Operator: "deploy approved; AI Developer go."
T+0:01: AI Developer self-attests Deploy phase.
T+0:02: Reads approval artifact ✓
T+0:03: Runs fusebase deploy. Files upload.
T+0:04: Runs post-deploy probe. Probe FAILS (regression).
T+0:05: Subagent halts; main agent: "deploy probe failed: <details>"
T+0:05+: Operator's attention is elsewhere (different tab, slack, lunch).
T+10:00: Operator returns; sees probe-failure message.
T+10:00+: First user-impact reports start coming in.
```

### Same flow with DP.6 in v2.1.1

```
T+0:    Operator: "deploy approved; AI Developer go."
T+0:01: AI Developer self-attests Deploy phase.
T+0:02: Reads approval artifact ✓ (DP.1)
T+0:03: Final worker-undisturbed re-check ✓ (DP.2)
T+0:04: Asks: "Type APPROVE-DEPLOY-NOW to proceed."  ← DP.6 pause
T+0:08: Operator types APPROVE-DEPLOY-NOW.            ← human at keyboard
T+0:09: Runs fusebase deploy. Files upload.
T+0:10: Runs probe. Probe FAILS.
T+0:11: Subagent halts; surfaces failure.
T+0:11+: Operator is RIGHT THERE — sees the red text in the same terminal.
T+0:12: Operator initiates rollback or diagnosis immediately.
```

### Updated Deploy phase ownership

```diff
| Pre-deploy | Verify approval artifact exists (DP.1) |
| Pre-deploy | Run final worker-undisturbed re-check (DP.2) |
+ | Operator confirm (DP.6) | STOP. Ask the operator to type the literal `APPROVE-DEPLOY-NOW` phrase. Anything else aborts. |
| Deploy | Execute the deploy command from the deploy handoff |
| Capture | Capture deploy hash (DP.3) |
| Verify | Run all probes + smoke prompts |
| Surface | If any probe / smoke fails: STOP (DP.5) |
| Hand back | Return deploy hash + probe results |
```

DP.6 is added to:

- `skills/role-discipline/SKILL.md` Deploy phase don't-list (with refusal phrasing for the "just deploy, I'm watching" violation request).
- `agents/ai-developer/AGENT.md` Deploy phase ownership table, don't-list (now `DP.1..DP.6`), and stop-conditions table.
- `workflows/greenlight-deploy.md` procedure step 4. Self-attestation phrase updated `DP.1..DP.5` → `DP.1..DP.6`.

### Why a magic phrase, not a yes/no?

Reflexively clicking "yes" is too easy. Typing `APPROVE-DEPLOY-NOW` requires attention. Same pattern as the existing `APPEND-ONLY` confirm in `install.sh` for protected-file detection. ~5 seconds of friction; trivial overhead for production safety.

## Hard rails — updated

| Risk | First-line guard (prompt) | Second-line guard (framework) |
|---|---|---|
| PO runs mutating shell command | PO Denied table forbids direct Bash; PO instructed to use wrapper only | **Wrapper** structurally rejects non-allowlisted subcommands |
| AI Dev runs deploy without human attention | DP.6 in role-discipline + AI Developer AGENT.md | **Magic-phrase confirm** structurally requires literal `APPROVE-DEPLOY-NOW` |

The `policies/command-policy.yml` deny-list and `policies/required-artifacts.yml` gates from earlier versions still apply.

## Upgrade notes

- **PO sub-agent users:** restart any session started before this upgrade so the wrapper-only-Bash rule loads from the new prompt.
- **Deploy automation:** DP.6 applies only to sub-agent / role-attested deploys. Operator-attested manual deploys (operator types `fusebase deploy` directly with no role attestation) are not subject to DP.6 — that's the operator's own choice and outside the agent's authority. CI/CD that auto-deploys is also out of scope (it isn't operating under role attestation).
- **Architectural shape unchanged:** still two sub-agents (PO + AI Developer), still parameterized by handoff filename, still operating under role-discipline. v2.1.1 hardens v2.1.0; it does not redesign it.

## Acknowledgement

Both refinements originated from external v2.1.0 evaluation feedback that surfaced specific failure modes (PO drift via shell injection; missed probe failures during prod cutover). Filed as bug-fix-shaped patches rather than backlog items because both could be implemented in <2 hours of focused work and both deepen the framework's defense-in-depth. Adopted in spirit and in the suggested implementation shapes (wrapper script + magic-phrase pause).
