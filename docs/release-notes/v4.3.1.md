# Fusebase Flow v4.3.1 — FR-25 `--write-baseline` clobber guard

**Released:** 2026-07-10
**Type:** patch (security hardening; closes a v4.3.0 deferred residual)

v4.3.0's adversarial review flagged, and the release deferred, a residual footgun in the FR-25 adoption path: `hooks/shared/module_size.py --write-baseline` writes the baseline to whatever `baseline_file` the **worktree** `policies/module-size.yml` names. A tampered worktree policy (e.g. `baseline_file: policies/approval-policy.yml`, editable via Bash, which the tool-time FR-07 hook does not gate) could therefore make `--write-baseline` overwrite an arbitrary file with FR-25 baseline text — recoverable via `git checkout`, but a real DoS.

**Fix:** `--write-baseline` now refuses (exit 2, fail-closed) to write when the target file already exists and is **not** a module-size baseline — i.e. its content does not start with the `# FR-25 module-size baseline …` header. A first write (target absent) or a legitimate re-write of an existing baseline (header present) proceeds unchanged, so the operator adoption path is unaffected.

This composes with v4.3.0's mint-scope fix — that closed the *self-approval* half (the FR-07 approval is minted only after affirmatively verifying, HEAD-derived and env-passed, that the baseline is the sole staged protected path); v4.3.1 closes the *file-overwrite* half. Verified by `hooks/tests/test-module-size.sh` scenario S11 (redirect to an existing config file → refused, file left intact).

No other behavior changed.
