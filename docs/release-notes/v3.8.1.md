# Fusebase Flow v3.8.1 — lossless first preserve-aware upgrade (U9)

**Release date:** 2026-06-01
**Previous version:** 3.8.0

## Summary

A downstream upgraded 3.7.0 → 3.8.0 and confirmed the U1 `FLOW:PRESERVE` protection works **going forward** — but the *transition itself* reset operator project-values **once**, because the pre-3.8.0 block had no `FLOW:PRESERVE` markers for the carry-forward to match. v3.8.1 closes that one-time gap: the first preserve-aware upgrade is now lossless too.

## Fixed

### U9 — seed the preserve region from a legacy project-values table

`refresh_overlay_block()` (in `hooks/local/post-fusebase-update.sh`) now handles three cases:

1. **Live block already has `FLOW:PRESERVE`** → carry the region forward (U1, 3.8.0).
2. **Live block is marker-less but has a `### Project-specific values` table** (a pre-3.8.0 block) → **seed** the new preserve region from that legacy table: detect it by its heading + "…project-specific rules win." footer, wrap it in the template's `FLOW:PRESERVE` markers, and carry it into the refreshed block. The operator's values survive the very first preserve-aware upgrade, and the block gains the markers so every subsequent upgrade is protected by U1.
3. **Neither** → raw template (byte-exact, as before).

AGENTS-only (CLAUDE has no preserve region). The v3.7.0 F2 byte-exactness lock, U1 (preserve), and U7 (legacy `---`) all still pass.

## Note for downstreams already on 3.8.0

If you upgraded 3.7.0 → 3.8.0 *before* this and re-applied your project values by hand, you're already protected — your block now has the `FLOW:PRESERVE` markers, so U1 keeps your values on every future upgrade. v3.8.1 only matters for installs making the pre-3.8.0 → ≥3.8.1 hop for the first time.
