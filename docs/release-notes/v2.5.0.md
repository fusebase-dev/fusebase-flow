# Fusebase Flow v2.5.0 — handoff prelude templates + role rename

**Released:** 2026-05-10

## TL;DR

Two operator-quality-of-life improvements that ship together:

1. **New `templates/handoff-implement.md` and `templates/handoff-deploy.md`** — handoff files now carry a role-bootstrap prelude at the top, so operators don't have to retype role-declaration prompts every time they open a fresh chat. Cross-IDE: works in Claude Code, Codex, Cursor, anything that reads markdown.
2. **Role rename: "Implementer" → "AI Developer"** across all framework narrative text. The agent identifier was always `ai-developer`; this consolidates the human-readable terminology.

No engine changes. No deferral mechanism changes. No `hooks/local/` scripts changed. **Pure framework + template + documentation work.**

## Who needs to upgrade

| Environment | Action |
|---|---|
| Projects on v2.4.1 with active development | Upgrade recommended for cross-IDE handoff convenience and terminology consistency. Zero-risk: no engine code changes. |
| Projects on v2.4.x with rare PO/AI Developer cycles | Optional. The "Implementer" → "AI Developer" rename is mostly cosmetic; existing handoffs continue to work. |
| Projects on pre-v2.4.x | Upgrade through the normal multi-version path (see `docs/fusebase-health/OPERATOR-WORKFLOWS.md` Scenario 14). v2.5.0 is no harder to land than v2.4.1. |

## What's new — the handoff prelude approach

### The problem

Before v2.5.0, every fresh AI Developer or Deploy chat required the operator to manually compose + paste a role-declaration prompt. That created friction at every fresh-session boundary, and three workarounds were considered:

| Option | Cross-IDE? | New infrastructure? | Operator typing per session |
|---|---|---|---|
| Slash command (`/implement <slug>`) | ❌ Claude Code only | New `.claude/commands/` file | `/implement <slug>` |
| SessionStart hook auto-hint | ❌ Claude Code only | Hook handler + state file | Nothing (auto) |
| **Handoff prelude (chosen)** | **✅ Any markdown reader** | **None — template change only** | **`Execute <path>`** |

The handoff-prelude approach won because it's the only one that works in Codex (which the project actively uses) without provider-specific machinery.

### The fix

PO sessions now author handoffs from canonical templates that have the role-bootstrap prelude built in:

```markdown
# Implement handoff — backlog-15 (2026-05-10)

## Role bootstrap (read this BEFORE any other reads)

You are operating as the **AI Developer** under Fusebase Flow v2.1.

**Self-attest in your first response, verbatim:**

> "Operating as AI Developer under Fusebase Flow v2.1. I will follow FR-01 through FR-15..."

**Hard invariants (do NOT violate):**
- Stop at the verification gate. Do NOT proceed to deploy.
- One task = one commit (FR-03).
- ...

## Mandatory pre-execution reads (in order)
1. `FLOW_RULES.md`
2. `AGENTS.md`
...

## Ticket header
| Field | Value |
| ... | ... |

## (rest of handoff body)
```

Operator's per-session typing burden drops to: `Execute docs/handoff/2026-05-10-backlog-15-implement.md`.

### Files added

- `templates/handoff-deploy.md` — AI Developer Deploy-phase handoff template (~150 lines, includes DP.6 magic-phrase invariant + DP.1 approval-artifact check)

## What's new — the "Implementer" → "AI Developer" rename

### Why

The agent file was always `agents/ai-developer/AGENT.md` (and mirrored at `.claude/agents/ai-developer.md`, `.codex/agents/ai-developer.md`). But narrative text used "Implementer" — a synonym for the same role. The split was confusing for new operators ("which one do I attest as?") and required mental translation between machine-readable identifier and human-readable noun.

v2.5.0 commits to "AI Developer" everywhere narrative text appears. The identifier `ai-developer` stays the canonical machine-readable form.

### Backward compatibility

In-flight tickets continue to work. The AI Developer role recognizes the older "Implementer" attestation as equivalent — older handoff files don't need to be re-authored. Future handoffs use the new templates and naturally pick up the new language.

## Upgrade

Same path as any v2.x → v2.x upgrade:

```bash
cd .fusebase-flow-source && git pull origin main && cd ..
bash hooks/local/upgrade-engine.sh
```

`upgrade-engine.sh` syncs `hooks/local/` files. The framework content (`workflows/`, `templates/`, `skills/`, `agents/`, `FLOW_RULES.md`, etc.) is auto-discovered by the engine — no separate sync step needed for those.

For projects that maintain a copy of `AGENTS.md`/`CLAUDE.md` overlay blocks (most projects do), run the recovery script after the engine sync to refresh the overlay blocks with the new "AI Developer" language:

```bash
bash hooks/local/post-fusebase-update.sh
```

The recovery script is additive + idempotent. It will detect that overlay block content has changed and re-append from the updated overlay templates.
