# Fusebase Flow v2.3.1 — cosmetic diff-count display fix in `upgrade-engine.sh`

**Released:** 2026-05-10

## What's new

Single-line cosmetic fix in `hooks/local/upgrade-engine.sh`. When the script preview displayed pending changes, files with line differences rendered with a broken count:

```
# Pre-v2.3.1
  • hooks/local/fusebase-flow-health-check.sh (200
0 line diffs)

# v2.3.1
  • hooks/local/fusebase-flow-health-check.sh (200 line diffs)
```

Functional logic was already correct in v2.3.0 — the upgrade itself worked fine. Only the count display was malformed.

## Root cause

`upgrade-engine.sh` runs with `set -euo pipefail`. The pre-fix line:

```bash
diff_count=$(diff "$src" "$f" 2>/dev/null | grep -cE "^[<>]" || echo 0)
```

interacts with pipefail in a subtle way:

1. `diff "$src" "$f"` exits 1 when files differ
2. `grep -cE "^[<>]"` writes the count to stdout (e.g. "200"), exits 0 (matches found)
3. `pipefail` makes the pipe's exit status the highest non-zero — so the pipe exits 1 (from `diff`)
4. `|| echo 0` fires because the pipe exited non-zero, appending "0\n" to stdout
5. `$(...)` captures BOTH the real count AND the appended "0", separated by a newline
6. The variable `diff_count` becomes `"200\n0"` (literal newline)
7. When echoed inside the change list, it renders across two lines

## The fix

```bash
diff_count=$(diff "$src" "$f" 2>/dev/null | grep -cE "^[<>]" 2>/dev/null || true)
```

`grep -c` always writes the count to stdout (even when count = 0), so swallowing the non-zero exit with `|| true` doesn't lose data. Inline comment added to the script explaining the pipefail interaction so future maintainers don't accidentally re-introduce the bug.

## Discovered during validation

This bug was caught during the v2.3.0 end-to-end smoke test in a downstream project. The upgrade succeeded; the dry-run preview rendered with a line break in the diff count display. v2.3.1 ships within hours of v2.3.0 — demonstrating that **always validating new releases against a real downstream upgrade scenario** (not just unit-testing in isolation) catches edge cases that pure code review misses.

The fix itself is one character (`echo 0` → `true`), but the discovery required actually running the script in production conditions with `set -o pipefail` active and a real diff happening.
