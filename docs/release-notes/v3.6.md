# Fusebase Flow v3.6.0 — upgrade-path hardening

**Release date:** 2026-05-31
**Previous version:** 3.5.2

## Summary

The install path was already mature. Verified operator feedback from upgrading a live project off an older Flow surfaced the remaining gap: the **in-place upgrade path** — refreshing an *already-installed* overlay to a newer Flow — had eight rough edges. The worst was structural (the only "upgrade" tool bumped `VERSION` while leaving skills/rules stale, and a clean upgrade could read RED on a single-provider project). All eight were verified against the code and fixed under one spec (`docs/specs/upgrade-path-hardening/`, decisions F1–F8). No skills added or removed (still 23 canonical).

## Added

### F1 — `hooks/local/upgrade.sh` (the keystone)

The missing in-place **content** upgrade. `upgrade-engine.sh` only ever synced 3 engine scripts + `VERSION`, so on an upgrade `VERSION` advanced while `skills/`, `FLOW_RULES.md`, etc. stayed stale. `upgrade.sh` closes this with a deliberate order:

1. Refresh canonical content from `.fusebase-flow-source/`: `skills/ agents/ workflows/ policies/ templates/ FLOW_RULES.md` + framework `docs/*.md`.
2. Re-mirror (`mirror-skills.sh` + `mirror-agents.sh`).
3. Sync embedded version strings (F7).
4. Version-aware refresh of drifted AGENTS.md/CLAUDE.md overlay blocks (F2).
5. Bump `VERSION` **last** — so VERSION can never lead content.

Backups (`.pre-upgrade-<ts>`), `--dry-run`, `--auto-yes`. It deliberately does **not** touch `.claude/settings.json` or CLI-owned assets.

### F7 — `hooks/local/sync-version-strings.sh` (context-safe, full coverage)

Derives the live `Fusebase Flow vX.Y.Z` self-attestation/banner strings from the `VERSION` file. It rewrites **only** the two live phrasings — `under Fusebase Flow v<semver>` and `runs **Fusebase Flow v<semver>**` — across every surface a fresh agent reads:

- `agents/ai-developer/AGENT.md`, `agents/product-owner/AGENT.md` (+ the re-mirrored `.claude/` & `.codex/` copies)
- `workflows/*.md`, `templates/handoff-*.md`, `FLOW_RULES.md`
- `.github/copilot-instructions.md`, `.cursor/rules/*.mdc`
- AGENTS.md / CLAUDE.md / GEMINI.md and the overlay templates

Because it is **context-anchored** rather than a blanket semver replace, historical/provenance references are preserved verbatim — `Shipped … v2.3.0+`, `Available since v2.4.0`, `DEPRECATED (… v3.2.0)`, `v2 (… v2.7.0+)`. Dated history (CHANGELOG, release notes, handoffs, specs) is excluded from the scan entirely. After editing canonical `agents/` it re-mirrors so the provider copies and audit manifests stay consistent. This fixes the ~12 files that still self-attested `v3.5.0` under a 3.6.0 install — including the agent definitions, so a fresh Product Owner / AI Developer now attests the correct version.

## Fixed

### F2 — version-aware overlay refresh (marker-anchored, idempotent)

`post-fusebase-update.sh --refresh-overlays` compares a *present* AGENTS.md/CLAUDE.md Flow overlay block to the template and **replaces it (with a `.pre-refresh-<ts>` backup) when drifted**, instead of always skipping "already present". Recovery's missing→append behavior is unchanged.

The detection and replacement are anchored on the `CUSTOM:SKILL:BEGIN`/`END` markers, **not** the heading. Both overlay templates wrap their heading *inside* those markers, so a heading-anchored extract (block = heading→EOF) could never match the marker-wrapped template — the drift check was always true and each run re-appended the wrapper, duplicating `BEGIN` and unbalancing the block (it compounded, because `upgrade.sh` calls `--refresh-overlays` every run, and could confuse the CLI's own CUSTOM-block preservation on the next `fusebase update`). The fix:

- Compare the live marker-wrapped block (the `BEGIN` immediately above the heading → EOF) to the template's marker-wrapped block; replace only on a genuine difference.
- The CLAUDE.md overlay template is now wrapped in the same `CUSTOM:SKILL` markers as AGENTS.md (so it gains the same CLI custom-block preservation); a legacy marker-less block migrates to the wrapped form on first refresh.
- Refreshing an already-current block is a **verified no-op** — `BEGIN`/`END` stay balanced at exactly 1, and no backup is written. `test-cli-flow-recovery.sh` now asserts no-op-on-current, restore-on-drift, and idempotent-re-run (the bug had slipped through because no `--refresh-overlays` idempotency assertion existed).

### F3 — hook wiring is genuinely opt-in

`post-fusebase-update.sh` previously auto-merged Flow lifecycle hooks into `.claude/settings.json` — contradicting CLAUDE.md's "hooks are opt-in: nothing runs until you copy settings.json.example." Default recovery now leaves settings.json untouched and prints a loud notice; the merge runs only with `--wire-hooks`. CLI Stop hooks are still preserved when you opt in.

### F4 — no false `CLI_LAYER_DRIFT` for single-provider projects

`check-cli-flow-conflicts.sh` emitted per-skill `MISSING` for every absent CLI provider skill whenever the mirror dir existed — so a Claude-only / non-FuseBase-Apps project that never installed the 19 CLI skills got a RED "structurally damaged" verdict after a clean upgrade. Now:

- **0 of N present** → one benign INFO ("not installed — benign for non-FuseBase-Apps / single-provider projects").
- **all present** → OK.
- **partial** → genuine MISSING drift (unchanged).

Same logic for the CLI app-agents.

### F5 — plain-dir upstream accepted

`upgrade-engine.sh` (and `upgrade.sh`) FATAL-ed without `.fusebase-flow-source/.git`, but the install docs copy the clone *minus* `.git`. They now warn and fall back to VERSION-file comparison; a real `.git` clone still enables HEAD/diff reporting.

### F6 — `.pyc` hygiene

`upgrade.sh` scrubs stray `*.pyc` before its commit advice (the gitignore rule was already present).

## Upgrading to v3.6.0

From an installed Flow overlay, with an upstream copy at `.fusebase-flow-source/`:

```bash
bash hooks/local/upgrade.sh --dry-run   # preview the plan
bash hooks/local/upgrade.sh             # apply (backups written)
bash hooks/local/preflight.sh           # expect 0/0
bash hooks/local/fusebase-flow-health-check.sh   # expect HEALTHY
# optional — (re)wire Flow lifecycle hooks:
bash hooks/local/post-fusebase-update.sh --wire-hooks
```
