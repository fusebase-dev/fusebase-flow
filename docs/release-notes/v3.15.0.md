# Fusebase Flow v3.15.0 — FR-24 write-time discipline delivery

**Release date:** 2026-06-08
**Previous version:** 3.14.2

## Summary

Makes the framework's **write-time documentation rules actually reach the agent that's writing** — reliably, in any session. Triggered by a real consumer report (WorkHub Managed): after upgrading to v3.14.2, an operator-launched AI-Developer fix chain kept producing verbose, human-oriented comments because FR-22's carrier skill is *description-matched* and never loaded in that flow.

The fix generalizes the problem (FR-20 zoom-out): FR-22 is one symptom of a **class**. The write-time rules — **FR-09 (Mode B), FR-18 (supersede), FR-22 (comments), FR-23 (documentation budget)** — all only save context if they're in the writing agent's context *at write time*, and all shared the same delivery gap (FR-23, the documentation rule, exposed identically). So this ships **one systemic delivery mechanism**, not a per-rule patch.

## The principle (codified)

Human operators do **not** read development artifacts — code comments, specs, decisions, tasks, gates, handoffs, the business-logic *index*. These are AI-consumed → **optimized for AI agents only** (dense, tiered, pointers-not-duplication, tripwire-only comments). The **human-facing surface** stays human-readable and is out of scope: README + translated READMEs, CONTRIBUTING/SECURITY/LICENSE/PUBLISHING, AGENTS/CLAUDE/GEMINI onboarding, the opt-in `business-logic.md` narrative.

## Why not the alternatives

- **`mandatory_load` the comment-policy skill** (or one per rule) — rejected: a 3rd-plus always-on skill taxes *every* session including non-writing roles, self-contradictory for context-economy rules (this was already the locked decision from the v3.11.0 work). The role-scoped digest delivers the same in-context guarantee for writers at near-zero cost to non-writers.
- **A regex/lint comment gate** — rejected (unchanged): tripwire-vs-restate and tier judgement are semantic, not pattern-matchable.

## Known boundary

The always-on digest reaches **main sessions**, not Task **sub-agents** (they don't auto-load skills and `session_start` doesn't fire for them). Sub-agents are covered by the **delegation push-block** (now inlining the digest) — handled in `handoff-implement` + `task-delegation`, unchanged in principle.
