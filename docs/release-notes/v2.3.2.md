# Fusebase Flow v2.3.2 — engine + recovery edge-case fixes

**Released:** 2026-05-10

## What's new

Two bundled fixes for issues surfaced during real-world use of v2.2.x → v2.3.x:

1. **`upgrade-engine.sh` self-update count is now accurate** (was undercounting by 1)
2. **Health check no longer reports `BROKEN` when `.fusebase-flow-source/` is cleaned up post-install** (now correctly reports `HEALTHY` with informational note)

Both are small fixes (~30 lines total). Pure behavior corrections; no API or content changes.

## Fix 1 — `upgrade-engine.sh` self-update count off-by-one

### Symptom (pre-v2.3.2)

When `upgrade-engine.sh` upgraded itself, the apply-summary undercounted:

```
[upgrade-engine] Applied (1):    ← should be 2
  ✓ VERSION (2.3.0 -> 2.3.1)
# but `upgrade-engine.sh` was actually replaced too — just not tracked in APPLIED
```

End state was always correct (file on disk did get the new content) — only the display was off.

### Root cause

The script overwrites itself via `cp "$src" "$f"` mid-execution. On Windows + Git Bash, the running bash process (executing from in-memory script) loses the `APPLIED+=("$f")` accumulation for the self-target after the `cp` replaces the file. Cause is OS-specific behavior of self-modifying scripts.

### Fix

Restructured the script:

- **Self-update detection** (`SELF_NEW` + `SELF_CHANGED` flags) happens in a dedicated pre-loop block before `FILES_TO_SYNC` iteration.
- **Self-update apply** happens before the main loop, so APPLIED tracking is reliable.
- **`upgrade-engine.sh` removed from `FILES_TO_SYNC`** — handled exclusively in the dedicated path.
- **Apply-summary message** now reads `"$SELF_TARGET (self-update; new logic active on next run)"` — accurately notes that the running script is still OLD logic; new logic kicks in on next run.
- Diff-line counting extracted into a `count_diff_lines` helper for consistency across both detection paths.

## Fix 2 — Engine reclassifies missing-clone from BROKEN to OK

### Symptom (pre-v2.3.2)

When `.fusebase-flow-source/` was absent (the documented post-install-cleanup state per `install-fusebase-cli-project.md` and `install-existing-project.md`) AND root-level `skills/` / `agents/` were also absent (e.g. some install patterns put canonical content only into provider mirrors), the engine reported:

```
⚠ .claude/agents/: cannot determine expected agent set (no upstream clone, no local agents/)
Verdict: BROKEN (exit 2)
```

This forced `BROKEN` even when the actual project state was healthy by design.

Surfaced empirically during the `paperclip+hermes-v1` install (commit `f73e204`):

- Install brief Step 16 explicitly cleaned up `.fusebase-flow-source/` per Fusebase Flow's documented hygiene.
- Install brief Step 15 expected `HEALTHY`.
- v2.3.1 engine reported `BROKEN` instead — wrong prediction by the brief, caused by this over-classification.

### Fix

Reclassified `EXPECTED_X_COUNT == 0` from `LOCAL_BROKEN` to `LOCAL_OK` with informational language:

```bash
LOCAL_OK+=(".claude/skills/: count not verified (no .fusebase-flow-source/ clone available; re-clone to enable upstream comparison)")
```

The check is now genuinely informational. Verdict no longer flips to `BROKEN` on this state alone. Operator who wants the count back can re-clone `.fusebase-flow-source/` anytime — but it's no longer required for `HEALTHY`.

Same change applied to both the skills check and the agents check. Inline comments cite v2.3.2 + reference to install-cleanup discipline.
