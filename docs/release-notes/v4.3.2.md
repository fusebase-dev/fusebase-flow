# Fusebase Flow v4.3.2 — operator-gate friction removed framework-wide; mirror + FR-25 hardening

**Released:** 2026-07-10
**Type:** minor (framework-wide UX rule + two robustness/security hardenings)

## Why

v4.3.0 fixed the *deploy* gate so the operator authorizes in chat and the agent runs the commands — but that fix was never generalized. Consumers kept hitting the same friction on the **other** gates: the FR-25 module-size adoption and FR-07 protected-path bootstrap approval still printed "operator-run … never agent-initiated" and made the human run `bash hooks/local/…`, then `git commit`, then `… --consume`. This release makes the rule universal, and closes two robustness gaps that v4.3.1's review had deferred.

## 1. Operator Gate Protocol (the framework-wide rule)

New shared protocol in `flow-skills/role-discipline/SKILL.md` (every role loads it):

> The operator's ONLY gate/approval action is a decision expressed **in chat**. When the operator approves in chat, the **agent** runs every command the approval requires — on their behalf. The operator never types a gate command.

| Gate | Operator does (chat only) | Agent does (in the owning role) |
|---|---|---|
| Deploy — Full lane (DP.6) | types `APPROVE-DEPLOY-NOW` | **Deploy session** authors artifacts + deploys |
| Deploy — Lightweight lane (DP.12) | plain go-ahead ("ship it") | **AI Developer** records + deploys in the same pass |
| Flow-internals protected-path edit (FR-07) | OKs the edit in chat | mints the digest-bound bootstrap approval, commits, consumes |
| Module-size adoption (FR-25) | OKs adopting the baseline in chat | runs `--write-baseline`, commits, consumes |
| Any other approval-gated action | OKs it in chat | authors the artifact (`approve-local.sh <action>`), runs the command |

**Two invariants preserved (this is a wording change, not an enforcement change):**

1. **Keystroke ownership changes, not role authority.** Only the Deploy session runs a Full-lane deploy; only the AI Developer runs a Lightweight one; PO/Architect never perform deploy or side-effect commands. A gate approved in the wrong role routes to the owning role. (Closes a HIGH review finding.)
2. **Self-approval stays forbidden.** Acting with NO operator authorization — or minting/adopting on the agent's own initiative to dodge a block — is self-approval, still forbidden. The git-hook protected-path block, the §2 secret scan, and the `--no-verify` deny are unchanged mechanical safety, not operator rituals.

Reworded ~30 carriers: `FLOW_RULES.md` (FR-25 rule + implication), the `module-size-discipline` skill, `module_size.py` BLOCK stderr + baseline header, `check-module-size.sh`, `write-bootstrap-approval.sh`, `upgrade.sh`/`post-fusebase-update.sh`, `command_policy.py` deny reason, `policies/module-size.yml` + `protected-paths.yml`, both install guides, the health-check family (`RESUMING-IN-FRESH-SESSION`, `ARCHITECTURE`, `KNOWN-LIMITATIONS`, `CODE-REFERENCES`, `health-check-deferrals`, `fusebase-flow-health-check.sh`), and the Cursor/Copilot provider security rules. Filed problem-catalog entry `gate-command-operator-friction`.

## 2. Mirror-manifest robustness

`mirror-skills.sh` and `mirror-agents.sh` previously appended manifest rows per file with `>>` after truncating — so two concurrent runs could interleave into a **duplicated** manifest (the real v4.3.1 CI-drift incident: 71 dup rows the hash-based `--check` couldn't see). Both scripts now build the manifest in memory and write it **once** via a temp file + atomic `mv` rename (`$$`-suffixed temp; an EXIT trap sweeps a half-written temp). `mirror-skills.sh --check` now also flags **duplicate manifest rows** as drift, so a race can never pass silently.

## 3. FR-25 `--write-baseline` HEAD-derivation + fail-closed

- The write target is derived from the **HEAD-committed** policy, not the worktree — a worktree `baseline_file:` redirect can no longer even create a stray file (v4.3.1 closed only the clobber half). Containment rejects absolute, Windows-drive-relative (`C:foo`), UNC, and repo-escaping committed values, and falls back to the default.
- Write-mode now **fails closed** (exit 2) when the policy can't be loaded (worktree policy deleted, or PyYAML missing) — previously it no-op'd (exit 0) and the wrapper would stage/mint whatever baseline was on disk, so a deleted policy + a pre-edited baseline could get an FR-07 approval minted for an attacker-supplied amnesty. (Closes a HIGH review finding.)
- The wrapper (`check-module-size.sh`) now **consumes the engine's printed resolved-path marker** (`[module-size] baseline-path:`) instead of independently reparsing the policy — so the staged/mint target can never diverge from where the engine actually wrote.
- Tests: `hooks/tests/test-module-size.sh` S11a/b/c (redirect variants) + S12 (missing-policy fail-closed, no mint) → **21/21**.

## Review

Codex gpt-5.6-sol (xhigh) plus a 4-lens Opus-4.8-max / Fable-5 adversarial workflow. The safety-regression lens independently confirmed **no self-approval loophole** was introduced by the reword. Both HIGH findings (missing-policy mint; role-authority) and all MEDIUM findings were fixed and re-verified before ship. Verified: `test-module-size.sh` 21/21, `test-baseline-merge.sh` 15/15, mirror `--check` 0-drift, plus the full CI gate.
