# Fusebase Flow v2.7.1 — `AskUserQuestion` removed from PO (closes FR-16 conflict)

**Released:** 2026-05-10

## TL;DR

Resolves a behavior conflict between v2.6.0's FR-16 / Operator Relay Protocol and the pre-v2.6.0 PO agent definition. The PO can no longer use `AskUserQuestion` (modal popup) — it must present options as Mode A chat-text tables that the operator can scroll, copy, follow up on, or forward to other sessions. Other roles (AI Developer, Deploy phase, Architect) are unaffected.

No engine changes. No schema changes. No template changes. Three small role-discipline edits.

## The friction this fixes

Real-world friction observed in `paperclip+hermes-v1` deploy session, 2026-05-10:

The PO produced a 4-option choice for the operator using `AskUserQuestion`. The operator reported back:

> "It's not convenient to interact with [the popup] because it's hard to follow up with the questions. Also, it's not possible to copy and paste to the product owner or from the product owner to the AI developer. ... It is easier when it just writes the questions in a standard way in a chat, providing the options. In this case, a human operator can copy and paste it or follow up with the questions."

That's exactly the pattern FR-16 was supposed to enforce: **options should be scrollable, copyable, forwardable**. The popup tool kills all three. The PO agent definition was written pre-FR-16 and explicitly recommended `AskUserQuestion`; that recommendation went stale when FR-16 shipped.

## The fix

| Operator need (per FR-16) | Mode A chat text | `AskUserQuestion` modal |
|---|---|---|
| Scroll back to compare options | ✓ | ✗ — closes after click |
| Copy options into another session for context | ✓ | ✗ — uncopyable modal |
| Ask a follow-up before deciding | ✓ | ✗ — modal forces single answer |
| Preserve in conversation history | ✓ — text persists | ✗ — only the selected answer survives |
| Forward options to AI Developer / Deploy session | ✓ | ✗ |

v2.7.1 explicitly forbids `AskUserQuestion` for the PO role. **Four coordinated edits**, critically including both the machine-readable enforcement point and the human-readable documentation:

### Change 1 — `agents/product-owner/AGENT.md` YAML frontmatter `tools:` field (THE ACTUAL ENFORCEMENT POINT)

This is what Claude Code actually reads to decide which tools the sub-agent gets when invoked via `Agent({subagent_type: "product-owner"})`. Documentation tables alone wouldn't have enforced anything — they're for humans.

```diff
---
name: product-owner
description: Use this agent to lead the Fusebase Flow ticket lifecycle from ...
-tools: Read, Glob, Grep, Bash, Write, Edit, AskUserQuestion
+tools: Read, Glob, Grep, Bash, Write, Edit
---
```

After this edit, a sub-agent invocation can't call `AskUserQuestion` — the tool literally isn't in its grant list. The denial is structural.

### Change 2 — `agents/product-owner/AGENT.md` Allowed table (documentation)

Removed for consistency with the frontmatter:

```diff
-| AskUserQuestion | every clarify Q-and-A; recommendations with 2–3 options + tradeoff |
```

### Change 3 — `agents/product-owner/AGENT.md` Denied table

Added (with rationale):

> | `AskUserQuestion` (modal popups) | **Conflicts with FR-16 / Operator Relay Protocol (v2.7.1+).** Popups are uncopyable, can't be scrolled back, force a single answer with no follow-up window, and can't be relayed to other sessions. The PO's job is to present options as Mode A chat-text tables that the operator can copy, scroll, follow up on, or forward to the AI Developer / Deploy session. Use markdown tables with options marked ⭐ for the recommendation, not popup tools. Other roles (AI Developer, Deploy phase, Architect) may use `AskUserQuestion` for narrow non-relay cases — the restriction is PO-only. |

### Change 4 — `skills/role-discipline/SKILL.md`

Three updates:

1. **Operator Relay Protocol step 3** explicitly says "Mode A chat-text" and "never use modal popup tools (`AskUserQuestion` etc.)" with anti-pattern entry for popup tools.
2. **PO.10 don't-list entry** expanded to forbid popup tools.
3. **New PO.10 refusal phrasing** for the case where operator (or another agent) requests popup-style output:

   > "Per FR-16 + PO.10 (v2.7.1+), I can't use popup tools — they break copy/scroll/follow-up and can't be relayed to other sessions. Here are the options as a chat table you can copy and reply to."

## What this looks like in practice

**Before v2.7.1** (modal popup):

```
[ AskUserQuestion modal opens ]
Smoke path?
○ Operator runs S2+S3 themselves now (15-30 min)
○ Create one minimal trigger issue per agent (~5 min)
○ Accept partial-verification commit now
○ Rollback now
[ Submit ] [ Cancel ]
```

Operator clicks one. Modal closes. Options are gone from chat scrollback. Operator can't show the choices to a teammate or forward them.

**After v2.7.1** (Mode A chat text):

```markdown
### Smoke path

| # | Option | What happens | Trade-off |
|---|---|---|---|
| ⭐ A | Operator runs S2+S3 themselves now | Highest-fidelity verification; 15-30 min | Operator's time |
| B | Create one minimal trigger issue per agent | ~5 min; deploy ships marked PARTIAL/PENDING | Lower confidence; verification deferred |
| C | Accept partial-verification commit now | Spec stays DRAFT until S2+S3 manually run later | Verification debt |
| D | Rollback now | Revert to v0.3.10; re-plan with different verification path | Lose today's progress |

Reply with `A` / `B` / `C` / `D` (or describe a different path).
```

Operator can scroll, copy, screenshot, forward to another session, ask "wait, what does C mean exactly?" — all the things FR-16 was supposed to enable.

## Other roles unaffected

`AskUserQuestion` is **not** banned framework-wide. Only the PO is restricted. Other roles can still use it for narrow non-relay cases:

- **AI Developer** rarely needs operator input during T-execution (it just executes against the locked plan)
- **Deploy phase** uses the DP.6 typed magic phrase (`APPROVE-DEPLOY-NOW`), which is a typed phrase by design, not a modal — already aligned with FR-16
- **Architect** investigates and reports; doesn't typically use `AskUserQuestion`

If any of these roles legitimately needs a single-answer popup (e.g., "which port?"), the tool is available. The restriction targets exactly the friction surfaced in `paperclip+hermes-v1`: PO-side relay scenarios.
