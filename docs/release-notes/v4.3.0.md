# Fusebase Flow v4.3.0 — FR-25 delta-aware adoption + deploy-approval ergonomics + mirror determinism

**Released:** 2026-07-10
**Type:** minor (guardrail-behavioral changes; no breaking API)

Closes two consumer-reported upgrade-friction defects plus a CI-flake root cause. Every change went through a multi-model adversarial review (Codex `gpt-5.6-sol` xhigh, Fable 5, Opus 4.8 at max effort) — which caught and forced fixes to **three** successive real security holes in the first-draft fixes (auto-mint over-authorization → its fail-open → a baseline-redirect), all closed and repro-verified before ship.

## FR-25 — adopting the module-size ratchet on an existing repo no longer hard-blocks

Before: a consumer whose repo predated FR-25 got their **first commit touching any pre-existing monolith hard-blocked**, and the "re-key the baseline" remedy self-collided with FR-07 (the baseline is a protected path) — a circular dead-end.

Now:
- **Delta-aware change gate** — a PRE-EXISTING over-ceiling file (over the ceiling at HEAD, not baselined) may be **touched or shrunk** in `--staged`/`--worktree` without blocking (the refactor path). Only files that **newly cross** the ceiling or **grow** while over it block. `--all` stays an absolute audit that reports every over-ceiling not-baselined file. A renamed-and-grown monolith still blocks (`--no-renames`); a symlink→regular-file typechange still blocks (`--diff-filter=ACMT`). The framework's own gating is unchanged (its two over-ceiling files are baselined).
- **Smooth, fail-closed adoption** — `check-module-size.sh --write-baseline` (operator-run) auto-mints the single-use FR-07 approval for the protected baseline and prints the commit → consume steps, so adoption is one sanctioned command (never `--no-verify`). It is **fail-closed**: it mints only after affirmatively verifying (via an env-passed, injection-proof scope check) that the baseline is the *sole* staged protected path, and it derives the baseline path from the **committed** policy (HEAD) so a worktree edit can't redirect the mint at another protected file.

## Deploy approvals — the agent stops making you run terminal commands after you've approved

Before: on a Full-lane deploy, even after the operator typed the DP.6 phrase `APPROVE-DEPLOY-NOW` and said "you run it", the agent refused and demanded the operator personally run `approve-local.sh …` — conflating **self-approval** (an agent authorizing its own deploy — forbidden) with **transcribing the operator's explicit authorization** (routine).

Now: once the operator types the DP.6 phrase, the **Deploy session authors every required approval artifact on the operator's behalf** — for ALL tickets (`production_deploy` + any `database_migration` / `auth_or_permission_change` / `protected_path_edit`). The operator's one action is the phrase.

The safety boundary is **preserved**, not removed:
- Authoring an approval **without** the phrase (Full) or a plain go-ahead (Lightweight) is self-approval and remains forbidden.
- The agent presents the **full scope** in chat before the phrase, and **an action not presented before the phrase is not covered by it** (informed consent).
- DP.6, DP.2 (FR-07 re-check), DP.10 (smoke), DP.11 (no delegated deploy) are unchanged. The approval **enforcement** (`command_policy.py`) is unchanged — it never checked the artifact's author; this is a discipline change that makes the Full lane consistent with the Lightweight lane (which already worked this way).

## Cross-platform mirror determinism

`mirror-skills.sh` and `mirror-agents.sh` now emit their checksum manifests under `LC_ALL=C sort`, so a Windows regeneration and Linux CI produce byte-identical row order — fixing the v4.2.2 mirror-drift CI flake at its root.
