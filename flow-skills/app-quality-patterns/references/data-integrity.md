# QP — Data integrity (loaded on trigger match; see ../SKILL.md)

| ID | Trigger | Requirement | Verify (copy-ready smoke recipe) | Anti-pattern |
|---|---|---|---|---|
| QP-10 | Any delete of an entity that other records reference | **Every delete names its cascade policy** — one of: cascade-delete children, block-while-referenced (with a user-facing reason), soft-delete/archive, or reassign — chosen in the spec, not improvised in code. No orphaned records after any delete path. | Create parent + ≥2 child/referencing records → delete parent via the UI → query the store for the children/references → assert the spec'd policy held and **zero orphans**; repeat via any second delete path (bulk, API). | Deleting the parent row only; children keep dangling foreign keys; counts/aggregates still include ghosts; orphan rows surface in other views as blank entries. |
| QP-11 | Any create/update/delete with the data visible elsewhere | **Mutations invalidate every dependent view.** After a mutation, all lists, counters, caches, and aggregates showing that data reflect it without a manual refresh. | Edit an entity in one view → navigate (without reload) to another view showing it (list, dashboard count) → assert updated value; then hard-refresh → assert persisted. | Stale list after edit; counter updates only on F5; cache key ignores the mutated entity. |
| QP-12 | Any submit/create action | **Double-submit is guarded.** The action disables (or is idempotent) while in flight; double-click, Enter+click, or a slow network cannot create duplicates. | Throttle network (slow 3G) → double-click create → assert exactly one record and a single success signal. | Two records from one user intent; button stays clickable during a 3s request. |
| QP-13 | Specs demanding instant feedback, or mutations that update the UI before server confirmation | **Optimistic updates roll back on failure, visibly.** If the server rejects, the UI restores the prior state AND surfaces an actionable error — never silent fake success. | Force a failing mutation (invalid payload / kill network) → assert UI reverted and an error with retry path appears; refresh → assert server state unchanged. | UI keeps the optimistic value after a 500; error only in console. |
| QP-14 | Any irreversible or bulk destructive action (delete, archive-all, overwrite, send) | **Destructive actions confirm scope or offer undo.** Irreversible ops require an explicit confirm that names the scope ("Delete 37 tickets?"); bulk actions preview what's affected; where feasible, prefer undo (grace window) over confirm dialogs. | Select multiple records → trigger bulk delete → assert a confirm naming the exact count/scope (or an undo affordance after); confirm → assert only the previewed records affected. | One-click hard delete; confirm dialog that says "Are you sure?" without scope; bulk action affecting records outside the visible selection. |
